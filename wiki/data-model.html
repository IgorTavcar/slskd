Data Model & Types

This page is the consolidated type catalog for the slskd domain model — every entity and enumeration the slskd HTTP/SignalR API emits, with field tables and representative JSON. It is the canonical reference that the Search, Transfers, Users & Rooms and Shares pages link into. The wire-level encoding of these same concepts (message codes, byte layout, attribute tuples) is covered in Soulseek Wire Protocol; this page describes the decoded, JSON-shaped form.

Serialization conventions

All examples reflect the actual JSON slskd produces. Three rules apply uniformly, configured once in Program.cs:

  • camelCase property names. ASP.NET Core's default JsonNamingPolicy.CamelCase is in effect. A C# property BytesTransferred serializes as bytesTransferred. The slskd web client (src/web/src/lib/*.js) consumes camelCase exclusively, confirming the convention.
  • Enums serialize as strings. A JsonStringEnumConverter is registered for both the REST pipeline and the SignalR JSON protocol. TransferDirection.Download serializes as "Download", not 0. For [Flags] enums (notably TransferStates) combined flags serialize as a comma-separated string, e.g. "Completed, Succeeded".
  • Null properties are omitted. DefaultIgnoreCondition = WhenWritingNull means a field that is null does not appear in the payload at all. Treat the absence of a key as "null/not provided".

File

A File represents one shared file as returned in a search result. slskd defines its own projection (slskd.Search.File) that flattens the audio metadata decoded from the wire-level attribute tuples into discrete, nullable fields. The originating Soulseek.NET File carries a raw attributes collection and attributeCount; slskd's search projection drops those raw fields and keeps only the decoded values.

Two flavours of File

The JSON shape of a file depends on the endpoint:

  • Search results (Response) embed slskd.Search.File — the decoded form documented below. No attributes[] / attributeCount keys.
  • Browse / directory contents (Directory) embed the Soulseek.NET File directly. That object additionally carries attributes (an array of {type, value}) and attributeCount alongside the same decoded bitRate/length/… fields. See File attributes.
FieldJSON nameTypeDescription
FilenamefilenamestringThe full remote filename exactly as it appeared on the wire, including the remote path. Uses the peer's native path separators (typically \ on Windows-origin shares).
Sizesizeint64File size in bytes.
ExtensionextensionstringThe file extension (may be empty). Sent verbatim by the remote peer; not necessarily derivable from the filename.
Codecodeint32A protocol "file code" field carried in search/browse responses. Conventionally 1 for a regular file; not otherwise interpreted by slskd.
BitRatebitRateint32?Decoded bit rate in kbps, if the peer supplied a BitRate attribute. Omitted when null.
Lengthlengthint32?Decoded duration in seconds, if a Length attribute was supplied. Omitted when null.
SampleRatesampleRateint32?Decoded sample rate in Hz (e.g. 44100), if a SampleRate attribute was supplied. Omitted when null.
BitDepthbitDepthint32?Decoded bit depth in bits (e.g. 16, 24), if a BitDepth attribute was supplied. Omitted when null.
IsVariableBitRateisVariableBitRatebool?VBR flag, decoded from the VariableBitRate attribute (true = VBR, false = CBR). Omitted when the attribute is absent.
IsLockedisLockedboolWhether this file is "locked" — i.e. shared but restricted by the remote peer (see Response). Note: in a Response, locked files are also segregated into lockedFiles[]; this per-file flag exists on the type but the segregation by collection is the primary signal.

Presence/absence of the decoded audio fields is the canonical way to distinguish lossy from lossless results. The web client treats a file as lossless when both sampleRate and bitDepth are present, and as lossy otherwise.

{
  "filename": "@@abcde\\Music\\Artist\\Album\\01 - Track.flac",
  "size": 31457280,
  "extension": "flac",
  "code": 1,
  "sampleRate": 44100,
  "bitDepth": 16,
  "isLocked": false
}

A lossy (MP3) result, where bit-rate fields are present and the lossless fields are absent:

{
  "filename": "@@abcde\\Music\\Artist\\Album\\01 - Track.mp3",
  "size": 8388608,
  "extension": "mp3",
  "code": 1,
  "bitRate": 320,
  "length": 214,
  "isVariableBitRate": false,
  "isLocked": false
}

File attributes & FileAttributeType

On the wire, audio metadata is not a set of named fields — it is a list of (type, value) integer tuples appended to each file entry (see Soulseek Wire Protocol → File attributes). Each tuple is a FileAttribute: a type drawn from the FileAttributeType enum and an int32 value. Soulseek.NET decodes these into the discrete bitRate/length/sampleRate/bitDepth/isVariableBitRate fields; slskd's search projection keeps only the decoded form, while browse responses expose both.

FileAttribute fieldJSON nameTypeDescription
TypetypeFileAttributeTypeThe attribute kind (serialized as a string, e.g. "BitRate").
Valuevalueint32The attribute's numeric value, interpreted per its type.
FileAttributeType memberWire valueUnit / meaningDecodes to
BitRate0Bit rate, in kbps.bitRate
Length1Length, in seconds.length
VariableBitRate2VBR flag; 0 = constant (CBR), 1 = variable (VBR).isVariableBitRate
SampleRate4Sample rate, in Hz.sampleRate
BitDepth5Bit depth, in bits.bitDepth
Note

The wire value 3 is unassigned in FileAttributeType (values skip from 2 to 4). The decoded fields exposed by slskd are derived by Soulseek.NET; there is no field-level guarantee that a peer sends a complete or consistent attribute set — clients should treat every decoded audio field as optional.

A browse-response file, which exposes the raw attribute tuples in addition to the decoded fields:

{
  "filename": "Album\\01 - Track.mp3",
  "size": 8388608,
  "extension": "mp3",
  "code": 1,
  "attributeCount": 3,
  "attributes": [
    { "type": "BitRate", "value": 320 },
    { "type": "Length", "value": 214 },
    { "type": "VariableBitRate", "value": 0 }
  ],
  "bitRate": 320,
  "length": 214,
  "isVariableBitRate": false
}

Response (search response)

A Response (slskd.Search.Response, projected from Soulseek.NET SearchResponse) is one peer's answer to a search. A Search aggregates many of these. The token correlates the response to the originating search request.

Locked files

"Locked" files are files the responding peer shares but restricts — typically files placed behind privilege, private-share, or per-user permission rules on the remote side. The peer reports them separately so the requester knows they exist but cannot freely download them. slskd surfaces them in a distinct lockedFiles[] collection with its own lockedFileCount; ordinary, freely-downloadable files are in files[] / fileCount.

FileCountfileCountint32Number of unlocked files in files.
FilesfilesFile[]The unlocked (freely downloadable) result files.
LockedFileCountlockedFileCountint32Number of locked files in lockedFiles.

Transfer (Download / Upload)

Several fields are computed ([NotMapped]) and serialized but never stored: bytesRemaining, percentComplete, elapsedTime, remainingTime. TransferStateCategories

slskd groups these composites into query-friendly sets (it avoids bitwise HasFlag in SQL by enumerating exact values). The categories are: Successful = {Completed | Succeeded}; Failed = {Cancelled, TimedOut, Errored, Rejected, Aborted}, each OR'd with Completed; Completed = Successful ∪ Failed (∪ bare Completed); Queued = {Requested, Queued, Queued|Locally, Queued|Remotely}; InProgress = {Initializing, InProgress}. The web client's retry rule is "state contains Completed and is not Completed, Succeeded"; the cancel rule covers the InProgress/Requested/Queued states.
