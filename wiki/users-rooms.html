Users, Rooms & Chat

People and messaging on the Soulseek network: querying another user's profile, statistics, presence and IP endpoint; browsing their shares; joining and chatting in public and private rooms with tickers and operators; and exchanging private messages with per-message acknowledgement. This page documents the slskd REST surface (UsersController, RoomsController, ConversationsController), the DTOs they emit, and the underlying Soulseek server messages those operations map to.

Conventions & serialization

All endpoints on this page are versioned under /api/v0 and require a bearer token. They are declared with [Authorize(Policy = AuthPolicy.Any)], so any authenticated role satisfies them Any. When the daemon is running as a relay agent (Program.IsRelayAgent), every action on these controllers returns 403 Forbidden — users, rooms and chat are controller-only on the controller node.

ConcernBehavior
Property namingJSON bodies use camelCase (ASP.NET Core default). A C# property UploadCount serializes as uploadCount.
EnumsSerialized as PascalCase strings via JsonStringEnumConverter — e.g. "Online", "Out", not numbers.
Null fieldsDefaultIgnoreCondition = WhenWritingNull. Nullable properties that are null (e.g. self, slotsFree, operatorCount, owner) are omitted from output entirely.
Path parameters{username} and {roomName} are [UrlEncoded]; clients must percent-encode them. The reference web client uses encodeURIComponent.
Message bodiesPOST of a single message or room/username string is sent as a JSON-encoded string (e.g. body literally "hello world" including the quotes), bound with [FromBody] string.

Users

The Users domain exposes read-only information about other peers on the network. Behind each call slskd asks the Soulseek server or the peer directly, caches the result, and (for status/statistics) keeps it fresh by "watching" the user server-side. Blacklisted usernames are treated as 404 Not Found by every UsersController action (and as a silent no-op by private messaging — see below).

The composite User record aggregates the per-user data slskd caches. It is not returned directly by the user endpoints below (which return the individual Info / Status sub-objects), but it is what appears in the application state under users[].

Presence & the privileged flag

UserPresence (from Soulseek.NET) is a three-state enum. slskd surfaces it as a string in Status.presence and as status on room user lists. The underlying integer values follow the Soulseek server's status code (documented inline on UserDataResponse.Status as "0 = offline, 1 = away, 2 = online"), but on the wire of the slskd API the value is always the string form.

String valueServer codeMeaning
"Offline"0User is not connected to the server.
"Away"1User is online but marked away.
"Online"2User is online and active.
Note

The integer-to-string mapping (0/1/2) is taken from the documenting comment on UserDataResponse.Status. The enum itself is defined in Soulseek.NET, not in this repository. The privileged flag (isPrivileged) is independent of presence and reflects whether the server considers the user a privileged (donor) member.

Source: src/slskd/Messaging/API/DTO/UserDataResponse.cs; src/web/src/components/Users/User.jsx, src/web/src/components/Rooms/RoomUserList.jsx

User endpoints

GET/api/v0/users/{username}/infoAny

Returns the peer's Info (profile, slots, queue length). Requires a direct peer connection. 404 if the user is offline (UserOfflineException) or blacklisted.

GET/api/v0/users/{username}/statusAny

Returns the peer's Status (presence + privileged). Resolved from the server and cached. 404 if offline/blacklisted.

GET/api/v0/users/{username}/endpointAny

Returns the peer's IP endpoint as an IPEndPoint — the address and listen port the server has on file for that user. Serialized through a custom IPAddressConverter. 404 if offline/blacklisted.

// GET /api/v0/users/RJ/endpoint  →  200
{
  "address": "203.0.113.7",
  "port": 2234
}
GET/api/v0/users/{username}/browseAny

Browses the user's entire shared file tree. slskd opens a peer connection and issues a SharesRequest; the peer responds with its full browse response. Returns an array of Directory objects (see Data Model and Shares & Browsing). The progress of an in-flight browse is tracked by BrowseTracker and cleared 5 seconds after completion. 404 if offline/blacklisted.

// GET /api/v0/users/RJ/browse  →  200  (abbreviated)
[
  {
    "name": "@@abcde\\Music\\Jazz",
    "fileCount": 2,
    "files": [
      { "filename": "01 - Take Five.mp3", "size": 8123456, "extension": ".mp3", "attributes": [ ... ] },
      { "filename": "02 - Blue Rondo.mp3", "size": 9011234, "extension": ".mp3", "attributes": [ ... ] }
    ]
  }
]
GET/api/v0/users/{username}/browse/statusAny

Returns the progress (a decimal, 0–100) of an in-progress browse of the user, from BrowseTracker. 404 when no browse is currently tracked for that user.

POST/api/v0/users/{username}/directoryAny

Requests the contents of a single directory from the peer (a peer FolderContentsRequest) rather than the whole tree. Returns an array of Directory. Use this for lazy/on-demand folder expansion when a full browse is too large.

Body fieldTypeRequiredDescription
directorystringyesThe full remote directory path to enumerate. Empty/null → 400 Bad Request.

{ "directory": "@@abcde\\Music\\Jazz" }
//  →  200  (array of Directory, same shape as /browse)

Watching & presence tracking

slskd maintains a non-expiring per-user cache (UserService.UserDictionary) of statistics, status and group, plus a WatchedUsernames set of users that have been registered server-side for live updates. The relationship matters for any interoperating client that wants fresh presence:

  • Watched users receive push updates from the server when their status changes; cached data for them is assumed current.
  • Watching is performed with WatchUserAsync (server WatchUser). The Soulseek server does not automatically send status and statistics on initial watch, so slskd explicitly fetches both before reporting the user as watched (IsWatched stays false until populated).
  • Watching is idempotent; repeated calls only refresh status/statistics.
  • The cache grows monotonically over a session — a deliberate trade-off so that group, speed and presence are available on hot paths (queueing, governors) without a round trip.
Note

There is no public REST endpoint that adds a watch directly; watching is a side effect of the daemon's own operations (e.g. tracking transfer counterparties, room members). The users[] array in application state reflects the current tracked set.

Source: src/slskd/Users/IUserService.cs; src/slskd/Users/UserService.cs

Rooms

Rooms are server-hosted chat channels. A client lists rooms advertised by the server, joins (which makes the server start streaming that room's messages and membership), reads cached messages/members, sends messages, sets a ticker, and for private rooms manages membership. slskd tracks every joined room in RoomTracker; the controller reads from that tracker rather than re-querying the server on each request.

Room endpoints

GET/api/v0/rooms/availableAny

Queries the server's room list (GetRoomListAsync) and returns a flat array of RoomInfoResponse. The server returns three buckets — public, private, and owned — which slskd flattens, tagging private rooms with isPrivate and owned rooms with isOwned. Each entry's isModerated is set from the server's moderated-room-names list.

GET/api/v0/rooms/joinedAny

Returns the array of room names currently joined (the keys of RoomTracker.Rooms) — a string[], not full room objects. Fetch a specific room's detail with the next endpoint.

GET/api/v0/rooms/joined/{roomName}Any

Returns the full RoomResponse for one joined room (membership + message history). 404 if not joined.

POST/api/v0/rooms/joinedAny

Joins a room. Body is the room name as a JSON string. Returns 201 Created with the new RoomResponse. If already joined, returns 200 OK (documented as 304 in source attributes but implemented as Ok()). If the server rejects the join (RoomJoinForbiddenException, e.g. a private room you are not a member of), returns 403 Forbidden with an explanatory message.

// POST /api/v0/rooms/joined
"nicotine"
//  →  201  (RoomResponse)
DELETE/api/v0/rooms/joined/{roomName}Any

Leaves a room and removes it from the tracker. 204 No Content on success, 404 if the room was not joined.

GET/api/v0/rooms/joined/{roomName}/usersAny

Returns the current member list as an array of UserDataResponse. The logged-in user's own entry carries self: true. 404 if not joined.

GET/api/v0/rooms/joined/{roomName}/messagesAny

Returns the cached message history (array of RoomMessageResponse) accumulated since the room was joined. The history is in-memory and starts empty at join time. 404 if not joined.

POST/api/v0/rooms/joined/{roomName}/messagesAny

Sends a chat message to the room (SendRoomMessageAsync). Body is the message as a JSON string. 201 Created on success, 404 if not joined.

// POST /api/v0/rooms/joined/nicotine/messages
"hello everyone"
//  →  201
POST/api/v0/rooms/joined/{roomName}/tickerAny

Sets the room ticker (SetRoomTickerAsync). Body is the ticker text as a JSON string. 201 Created, 404 if not joined. See Tickers.

POST/api/v0/rooms/joined/{roomName}/membersAny

Adds a member to a private room you own/operate (AddPrivateRoomMemberAsync). Body is the username to add, as a JSON string. 201 Created, 404 if the room is not joined. The server enforces ownership/operator authority; an unauthorized request will be rejected at the protocol layer.

Source: src/slskd/Messaging/API/Controllers/RoomsController.cs; src/web/src/lib/rooms.js

Tickers, private & owned rooms, operators

ConceptDescription
TickerA short room-wide status/banner line stored by the server and shown to all members. Set via the /ticker endpoint. Ticker text is a single string; setting it replaces the previous value for the setter.
Public roomisPrivate = false. Anyone may join. No owner, operators or member list management.
Private roomisPrivate = true. Membership is controlled. Only members may join; non-members get 403 on join. Has an owner and a set of operators.
Owned roomA private room whose owner is the logged-in user (isOwned = true in the available list). The owner can add members (/members) and appoint operators.
OperatorsUsers with elevated rights in a private room. Reported in operators[] with an operatorCount. Operator/owner appointment beyond "add member" is a server-side concern not surfaced as distinct slskd endpoints.
Moderated roomisModerated = true when the room name is in the server's moderated-room list for the logged-in user.
Note

operatorCount, operators and owner are populated only for private rooms; for public rooms they are null and therefore omitted from the JSON.

Private chat (conversations)

A conversation is slskd's persistent, per-user thread of private messages. Unlike room history (in-memory), private messages are stored in a SQLite database (MessagingDbContext), survive restarts, and carry an acknowledgement state. Inbound messages map to the server's private message delivery; outbound messages are sent and recorded locally.

Conversation endpoints

GET/api/v0/conversationsAny

Lists conversations. Query parameters:

Query paramTypeDefaultEffect
includeInactiveboolfalseWhen false, only isActive conversations are returned.
unAcknowledgedOnlyboolfalseWhen true, only conversations with unacknowledged messages are returned.

Returns an array of Conversation.

GET/api/v0/conversations/{username}Any

Returns one Conversation. Query param includeMessages (bool, default true) toggles inclusion of the messages[] array. 404 if no such conversation.

GET/api/v0/conversations/{username}/messagesAny

Returns just the message array (PrivateMessage[]) for a conversation. Query param unAcknowledgedOnly (default false) filters to messages where isAcknowledged == false. 404 if no such conversation.

POST/api/v0/conversations/{username}Any

Sends a private message. Body is the message as a JSON string. On success the message is sent over the network (SendPrivateMessageAsync) and stored locally as an outbound, already-acknowledged record. Returns 201 Created. An empty/null message → 400 Bad Request. If the recipient is blacklisted, the call is a silent no-op returning 200 OK (no message is sent).

// POST /api/v0/conversations/RJ
"thanks!"
//  →  201
PUT/api/v0/conversations/{username}Any

Acknowledges all messages from the user. 200 OK; 404 if no such conversation.

PUT/api/v0/conversations/{username}/{id}Any

Acknowledges a single message by its server id. 200 OK; 404 if the conversation or message id is not found.

DELETE/api/v0/conversations/{username}Any

Closes (marks inactive) the conversation. 204 No Content; 404 if no active conversation exists. The record is retained; the conversation re-activates if a new message arrives or is sent.

Source: src/slskd/Messaging/API/Controllers/ConversationsController.cs; src/web/src/lib/chat.js

Acknowledgement semantics

Soulseek requires each delivered private message to be acknowledged to the server, otherwise it will be re-delivered (replayed) on subsequent logins. slskd records inbound messages as unacknowledged and exposes acknowledgement through the PUT endpoints; under the hood acknowledgement calls AcknowledgePrivateMessageAsync(id) per message. Outbound messages are recorded as already acknowledged (they need no server ack) and stored with id = 0.

Tip

The reference web client acknowledges unread messages immediately on opening a conversation, then fetches the thread. A custom client that omits acknowledgement will see the same messages replayed (wasReplayed: true) every time it reconnects.

Source: src/slskd/Messaging/ConversationService.cs (AcknowledgeMessageAsync / AcknowledgeAsync); src/web/src/components/Chat/Chat.jsx

Server message mapping

The slskd operations above are convenience wrappers over Soulseek server messages handled by the embedded Soulseek.NET client. For clients speaking the native wire protocol directly, the relevant server message codes (see Soulseek Wire Protocol) are:

slskd operationServer messageNotes
Send private messageCode 22 MessageUserSender → server → recipient. The server assigns the message id seen by the recipient.
Acknowledge private messageCode 23 MessageAckedAcknowledges receipt by id; suppresses replay on next login.
Watch user / get statusWatchUser / GetStatus / GetPeerAddressStatus, statistics and endpoint resolution. slskd watches before relying on cached status.
Room listRoomListReturns public, private, owned and moderated buckets, flattened by slskd.
Join / leave roomJoinRoom / LeaveRoomJoin begins server-side streaming of room messages and membership.
Room messageSayInChatRoom / room message receivedSent via SendRoomMessageAsync; inbound surfaces via the room-message event.
Set tickerRoomTickerSetServer stores and broadcasts the ticker.
Add private room memberPrivateRoomAddUserOwner/operator only; server enforces authority.
Browse sharespeer SharesRequestPeer-to-peer, not server-mediated. See Shares.
Directory contentspeer FolderContentsRequestSingle-folder enumeration, peer-to-peer.
Note

Codes 22 (MessageUser) and 23 (MessageAcked) are stated in the page brief and correspond to the standard Soulseek server protocol; the exact numeric codes for room/watch/list messages are defined in Soulseek.NET rather than this repository — consult Soulseek Wire Protocol for the full catalog. The named operations above are verified from the controller call sites; the precise message names/codes for non-PM operations are noted as inferred from Soulseek.NET method names.

Real-time updates

slskd's ApplicationHub SignalR hub broadcasts two named messages — STATE and OPTIONS — and (on connect) sends the current state and options to the caller. The application State includes a rooms[] (joined room names) and users[] (tracked User records) snapshot, so changes to the joined-room set and tracked-user data propagate through the STATE broadcast.

Important

Room message history, room membership and private-message threads are not pushed over SignalR as discrete events. The reference web client obtains these by polling the REST endpoints: the rooms view polls the joined-room list every 500 ms and the active room's messages/users every 1 s; the chat view polls conversations every 5 s. An interoperating client should likewise poll the relevant endpoints (or watch the STATE broadcast for the room/user set) rather than expecting per-message push notifications.

Source: src/slskd/Core/API/Hubs/ApplicationHub.cs; src/slskd/Core/State.cs; src/web/src/components/Rooms/Rooms.jsx, src/web/src/components/Chat/Chat.jsx
