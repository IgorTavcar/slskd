Soulseek Wire Protocol

The native Soulseek protocol is a binary, little-endian, length-prefixed message protocol spoken over plain TCP. It spans three logical domains — a central server, direct peer connections, and a distributed search tree — plus a small connection-initialization handshake. slskd does not implement these bytes itself: it delegates the entire wire protocol to the Soulseek.NET library (v10.0.1). This page documents the on-the-wire format so that an engineer can implement a native client in any language, or understand exactly what slskd exchanges with the network underneath its REST/SignalR surface.

Provenance & accuracy

The Soulseek protocol is proprietary and undocumented by its authors. Every code and structure here is community reverse-engineered. The catalogs below are cross-checked against two authoritative sources: the Nicotine+ protocol documentation (the de-facto reference) and the MessageCode.cs enum of Soulseek.NET — the exact library slskd uses, so its naming is what slskd "knows". Where the two disagree on a name, both are shown. Codes marked obsolete/deprecated are documented for completeness but are not exercised by current clients. The protocol drifts as the official client evolves; treat any single code as verifiable-but-not-guaranteed.

Message framing

Every message — on every connection type — is a length-prefixed frame. The 4-byte length prefix counts the bytes that follow it (i.e. it excludes itself), and always includes the code field.

+----------------+----------------+--------------------------+
| length: uint32 | code           | payload (length - sizeof(code) bytes) |
+----------------+----------------+--------------------------+
       4 bytes     1 or 4 bytes              variable

The width of the code field depends on the domain:

DomainConnectionCode widthFrame
Serversingle long-lived TCP to the central serveruint32<uint32 length><uint32 code><payload>
Peer'P' connection, peer-to-peeruint32<uint32 length><uint32 code><payload>
Peer-initfirst message on any new peer connectionuint8<uint32 length><uint8 code><payload>
Distributed'D' connection, search treeuint8<uint32 length><uint8 code><payload>
File'F' connection— (none)no message framing after init; raw byte stream
Length includes the code

Because length counts the code field, a server/peer message with an empty payload has length = 4; a distributed/peer-init message with an empty payload has length = 1. A reader frames by: read 4 bytes (length, LE) → read exactly that many more bytes → the first 1 or 4 of those are the code.

Primitive types & encoding

All integers are little-endian. There is no alignment or padding; fields are packed back-to-back. The protocol uses a small set of primitives:

TypeWidthEncoding
uint8 / bool1 byteUnsigned byte. As a bool, 0x00 = false, 0x01 = true (any non-zero is true on read).
uint324 bytesLittle-endian unsigned 32-bit. Used for lengths, codes, tokens, ports, counts, IPv4 addresses, speeds.
uint648 bytesLittle-endian unsigned 64-bit. Used for file sizes and the file-transfer offset.
string4 + N bytes<uint32 length N><N bytes>. Bytes are UTF-8 in modern clients. (Historically Latin-1 / CP-1252; readers should be lenient.) Not NUL-terminated.
bytes4 + N bytesSame framing as string but raw/opaque payload (e.g. compressed blobs, user-info pictures).

Higher-level structures are built by concatenation. Two conventions appear constantly:

  • Lists — a uint32 count followed by that many repetitions of an element layout. (For example, a search result's file list, or a room list.)
  • IPv4 address — sent as a single uint32. Note the byte order: in GetPeerAddress / ConnectToPeer the address is read so that octet order is effectively reversed relative to the LE integer; implementers should validate against the reference rather than assume.

Message obfuscation

The base protocol is plaintext over TCP — there is no transport encryption. The official client and Nicotine+ support an optional message obfuscation scheme on peer/distributed connections, negotiated per-connection (obfuscation type: 0 = none, 1 = rotated). It is a lightweight rotating-XOR scheme intended only to evade naive ISP/DPI matching — not a security feature. It provides no confidentiality or integrity.

slskd / Soulseek.NET

Soulseek.NET (and therefore slskd) speaks the unobfuscated protocol. Interoperability does not require implementing obfuscation; peers fall back to plaintext. There is no transport-layer authentication or encryption on the Soulseek network — credentials are sent to the central server in the Login message (with a password hash; see below), but peer traffic is fully in the clear.

The four message domains

The protocol is partitioned by which connection a message travels over. Direction notation used below: C→S client to server, S→C server to client, peer peer-to-peer (either direction), distributed over the search tree.

uint32 code

Server

Login, status, rooms, private chat, search injection, the distributed-tree control plane.

 uint8 code

Peer-init

PeerInit & PierceFirewall — the two messages that bootstrap any peer/file/distributed connection.

 uint32 code

Peer

Browse, user info, folder contents, search results, transfer negotiation, queueing.

 uint8 code

Distributed

The broadcast tree that fans search requests out to every online peer.

Peer-init (connection) messages

These two messages are not in the peer message space — they have their own tiny uint8-coded numbering and are sent as the first message on a freshly-opened peer, file ('F') or distributed ('D') connection to declare what the connection is for.

Code 1 PeerInit direct

Sent by the connection initiator when it could reach the remote peer's listen port directly. Declares who you are and what the connection carries.

<uint32 length><uint8 code=1>
<string  username>        // the initiator's own username
<string  connectionType>  // "P" (peer), "F" (file), or "D" (distributed)
<uint32  token>           // typically 0 for a direct PeerInit
Code 0 PierceFirewall indirect

Sent by a peer that received a ConnectToPeer relay and is now connecting back to the requester (firewall traversal). Carries only the token that ties it to the original request.

<uint32 length><uint8 code=0>
<uint32 token>            // echoes the token from ConnectToPeer (code 18)

Codes: Soulseek.NET MessageCode.Initialization (PierceFirewall = 0, PeerInit = 1).

Compression

BrowseResponse (5), SearchResponse (9) and FolderContentsResponse (37) payloads are zlib (RFC 1950) compressed after the standard length/code prefix. A native implementer must zlib-inflate the payload before parsing the file list. The length prefix counts the compressed bytes.

Distributed messages

Soulseek's search is not flooded from the server to every client directly. Instead the server maintains a distributed broadcast tree: each client tries to attach to a parent and may accept children. The server injects each search at the tree roots, and it propagates down. Distributed messages travel over 'D' connections and use uint8 codes (Soulseek.NET MessageCode.Distributed).

CodeName (Soulseek.NET)DirPurposeKey fields

4BranchLevel (DistribBranchLevel)distributedThe sender's distance from the root. You adopt the sender as parent and set your level = theirs + 1.level (int32)
5BranchRoot (DistribBranchRoot)distributedThe username of the sender's branch root, propagated downward.rootUsername

Branch root, branch level & search flow

  • Branch root — a top-level node directly fed searches by the central server (via EmbeddedMessage, code 93). Its branch level is 0.
  • Branch level — a node's depth in the tree. On adopting a parent that sends BranchLevel(n), the child sets its own level to n + 1 and re-advertises that to its own children. A level of 0 means the sender is a branch root.
  • Parent selection — the server hands out candidate parents via NetInfo (code 102) when you advertise HaveNoParents (71). You connect to candidates; the first one that sends you a SearchRequest (with its BranchLevel/BranchRoot) becomes your effective parent.
  • Child capacity — bounded by ParentMinSpeed (83) and ParentSpeedRatio (84): roughly maxChildren = yourAvgSpeed / ratio. slskd additionally caps children via its own DistributedNetwork.ChildLimit option.
  • Search propagation — a node receiving a DistribSearch (3): (a) matches it against its own shares and, if it has hits, opens a fresh 'P' connection back to the searcher to deliver a SearchResponse (9); and (b) forwards the identical raw DistribSearch to each of its children. Results therefore return out-of-band via direct peer connections, never back up the tree.

Connection establishment

Because most users are behind NAT/firewalls, the protocol supports both direct and server-mediated ("indirect") peer connection setup. Every peer connection is typed: 'P' (peer messages), 'F' (file transfer), or 'D' (distributed). The dance below applies to all three types; only the type string in PeerInit / ConnectToPeer changes.

Direct path (initiator can reach the target)

  1. Initiator resolves the target's address via GetPeerAddress (server code 3) → gets ip:port.
  2. Initiator opens a TCP connection to that ip:port.
  3. Initiator sends PeerInit (init code 1) = { ownUsername, connectionType, token }.
  4. Connection is now established; domain messages flow.

Indirect path (direct connect fails — firewall traversal)

  1. Initiator picks a token and sends ConnectToPeer (server code 18) = { token, targetUsername, connectionType } to the server.
  2. The server relays a ConnectToPeer to the target, carrying the same token plus the initiator's ip:port.
  3. The target opens a TCP connection back to the initiator's ip:port and sends PierceFirewall (init code 0) = { token }.
  4. The initiator matches the token to its pending request; the connection is established and proceeds as if direct.
  5. Failure: if the target also cannot connect back, it sends CannotConnect (server code 1001) { token, username }, which the server relays to the initiator so it can give up.
Note

In practice a client races both paths: it sends PeerInit directly and requests an indirect ConnectToPeer, then uses whichever succeeds first (matching on the token). Soulseek.NET implements this race internally; slskd never sees the individual init messages — it only configures the listen endpoint and reacts to high-level connect/transfer events.

slskd configures the inbound listener at src/slskd/Application.cs:433–435 (listenIPAddress, listenPort, enableListener: true); default listen port 50300 (src/slskd/Core/Options.cs:1770) on 0.0.0.0 (Options.cs:1761). Outbound resolution uses ConnectToUserAsync (src/slskd/Transfers/Downloads/DownloadService.cs:346).

File transfer over an 'F' connection

A file download is negotiated over the 'P' connection, then the bytes flow over a separate 'F' connection keyed by a token. The roles are inverted from intuition: in Soulseek the uploader drives the transfer.

Negotiation

  1. Downloader sends QueueDownload (peer code 43) { filename } to the uploader (or a TransferRequest with direction = download).
  2. When the uploader has a free slot, it sends TransferRequest (peer code 40) { direction=1 (upload), token, filename, size }.
  3. Downloader replies TransferResponse (peer code 41) { token, allowed=true } (or allowed=false, reason).
  4. The uploader opens a new 'F' connection to the downloader. Its first message after PeerInit(type='F') is the bare uint32 token identifying which negotiated transfer this connection carries.
  5. The downloader then sends an 8-byte uint64 file offset (the resume point; 0 for a fresh download).
  6. The uploader streams the raw file bytes from that offset to EOF. There is no per-chunk framing — the 'F' connection is a plain byte pipe; completion is detected when size - offset bytes have been received.

Queue-position control

MessageCodeMeaning
PlaceInQueueRequestpeer 51Downloader asks "where am I in your upload queue?"

Offset is for resume, not range requests

The 8-byte offset is intended for resuming an interrupted download. Repeatedly reconnecting with different offsets to fetch arbitrary chunks is non-idiomatic and tends to be treated as a failed transfer by the uploader. Request the whole remaining file in one stream.

See Transfers & Queueing for how slskd models this (upload slots, queue forecasting, place-in-queue API). slskd drives downloads with Client.DownloadAsync (src/slskd/Transfers/Downloads/DownloadService.cs:1284) and queries position via GetDownloadPlaceInQueueAsync (src/slskd/Transfers/Downloads/DownloadService.cs:723).

Search delivery end-to-end

Putting the domains together, here is the full lifecycle of one global search from the searcher's point of view:

  1. Submit. Searcher picks a uint32 token and sends FileSearch (server code 26) { token, query } to the server. (A wishlist search uses code 103; a single-user search uses code 42; a room search code 120.)
  2. Inject. The server wraps the query as a DistribSearch and pushes it to the branch roots via EmbeddedMessage (server code 93 / distributed code 93), carrying the searcher's username and token.
  3. Propagate. Each node forwards the raw DistribSearch (distributed code 3) down to its children, fanning out across the whole tree.
  4. Match. Every peer matches the query against its own shares.
  5. Reply. A peer with hits opens a fresh 'P' connection back to the searcher (direct or indirect per Connection establishment) and sends SearchResponse (peer code 9) containing: username, the original token, the matched files[], a hasFreeUploadSlot flag, the responder's uploadSpeed (avg, server-reported), the queueLength, and a list of lockedFiles[] (results the searcher is not permitted to download). The payload is zlib-compressed.
  6. Correlate. The searcher matches each inbound response by token back to the original query and aggregates results until a timeout elapses.

When slskd is the responder, it builds the reply in SearchResponseResolver:

response = new SearchResponse(
    Client.Username,
    token,
    uploadSpeed: State.CurrentValue.User.Statistics.AverageSpeed,
    hasFreeUploadSlot: forecastedPosition == 0,
    queueLength: forecastedPosition,
    fileList: results);

Source: src/slskd/Application.cs:1635–1768 (SearchResponseResolver). Returning null (or an empty file list) suppresses the reply, so no response is sent for zero matches. When searching, slskd calls Client.SearchAsync(...) (src/slskd/Search/SearchService.cs:309) and collects SearchResponse objects. See Search.

How slskd maps onto the protocol

slskd does not implement any of the bytes above. It constructs a single SoulseekClient (Soulseek.NET) and registers high-level handlers; the library owns all framing, connection management, the distributed tree, NAT traversal and transfer plumbing. An application author building on slskd should use its REST API and real-time events rather than re-implementing the wire protocol — re-implement the protocol only if you are writing a fully native client without slskd.
