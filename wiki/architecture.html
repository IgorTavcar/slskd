Architecture & Concepts

The Soulseek platform is not a single protocol but three cooperating planes: a central coordination server, a mesh of direct peer-to-peer connections, and a distributed search-propagation tree. slskd is a long-running daemon that embeds a full Soulseek client (the Soulseek.NET library) inside an ASP.NET Core host, and exposes that client to your application through a REST and SignalR API. This page establishes the mental model a builder needs before reading any of the protocol or API references: what each plane does, how connections are established (including through firewalls), how slskd is structured internally, and the lifecycle of the server connection. It cites the slskd source throughout so every claim is traceable.

1. The big picture: three planes

Key insight

You submit a search to the central server; the server injects it into the distributed tree where it fans out to many peers; each matching peer opens a direct peer connection back to you to deliver its results. Three planes, one logical operation. See Search for the full sequence.

2. The central Soulseek server

The central server is the only well-known, fixed endpoint in the network. A client maintains exactly one TCP connection to it for the lifetime of a session. It performs coordination that cannot be done peer-to-peer: authenticating logins, tracking who is online, distributing room chat, and вҖ” critically вҖ” telling clients each other's IP address and port (peers are not directly addressable until the server reveals their address).

3. Peer connections (P / D / F)

All non-server traffic flows over direct TCP connections between two clients. The Soulseek protocol multiplexes three distinct purposes onto separate connections, each identified by a single ASCII letter sent during connection setup. slskd configures all three independently through SoulseekClientOptionsPatch.

TypeLetterPurpose
PeerPGeneral peer messaging: search replies, share browse, user info, folder-contents requests, and transfer negotiation (queue/place/start). One per peer pair, reused.
DistributedDDistributed-network links: the connection to a parent and to each child, over which search requests are propagated. See В§5.
File transferFA dedicated connection carrying the raw bytes of a single file transfer. One per active transfer.

Source: nicotine+ SLSKPROTOCOL (connection-type letters); src/slskd/Application.cs (peerConnectionOptions, distributedConnectionOptions, transferConnectionOptions passed to SoulseekClientOptionsPatch).

Outgoing vs incoming & the listen port

A client both initiates connections to other peers (e.g. to fetch a browse listing) and accepts incoming connections (e.g. from a peer delivering a search result). To accept incoming connections, the client must listen on a TCP port and advertise it to the central server via SetListenPort. slskd listens on 0.0.0.0:50300 by default.

SettingDefaultCLI flagEnv varYAML

Port forwarding matters

If the listen port is not reachable from the internet (no port-forward / firewall rule), other clients cannot open direct connections to you. The slskd documentation lists the symptoms explicitly: "poor search results and the inability to browse or retrieve user information for some users." Two firewalled peers cannot connect to each other at all (see В§4). Forwarding the listen port is the single highest-impact connectivity fix.

Source: src/slskd/Core/Options.cs (SoulseekOptions.ListenIpAddress = "0.0.0.0", ListenPort = 50300); src/slskd/Application.cs:459; docs/config.md (Listen IP Address and Port).

4. Indirect connections & firewalls

Because peers are not addressable until the server reveals their address, and because many peers sit behind NAT/firewalls, the protocol defines a server-brokered fallback for establishing a peer connection. The mechanism inverts the connection direction so that a firewalled peer dials out rather than being dialed.

  1. Direct attempt. The initiator resolves the target with GetPeerAddress (3) and attempts a direct TCP connection to the returned IP:port, sending a PeerInit message that declares the connection type (P/D/F) and a unique token.
  2. Indirect fallback. If the direct attempt fails (target firewalled), the initiator sends ConnectToPeer (18) to the central server, naming the target user, the connection type, and the same token.
  3. Server relay. The server forwards the request to the target, who then dials back to the initiator and sends a PierceFirewall message carrying the token, allowing the initiator to match the inbound connection to its pending request.
  4. Give up. If the target also cannot connect (e.g. both peers firewalled), it returns CantConnectToPeer (1001) to the server, which relays it to the initiator.
Note

The token correlates the original request, the ConnectToPeer relay, and the eventual PierceFirewall handshake вҖ” it is how a client distinguishes which pending operation an unexpected inbound connection belongs to. The exact byte layout of PeerInit, PierceFirewall, ConnectToPeer and CantConnectToPeer is documented in Soulseek Wire Protocol.

Source: nicotine+ SLSKPROTOCOL (ConnectToPeer / PierceFirewall / CantConnectToPeer); message codes verified against the spec's server-message catalog.

5. The distributed search network

Search requests are not broadcast by the server to every user вҖ” that would not scale. Instead, clients self-organize into a distributed tree. Each participating client maintains one connection upward to a parent and up to N connections downward to children. The server injects each submitted search near the roots; every node re-broadcasts the request to all of its children, so the query fans out across the whole network in a few hops.

Results do not return through the tree

7. Server connection lifecycle

The state of the server connection is modelled by the SoulseekClientStates flags enum from Soulseek.NET. Because it is a flags enum, several bits can be set at once (e.g. Connected | LoggingIn during the login handshake). slskd exposes the current value through the ServerState DTO and derives convenience booleans from it.

State flagMeaning
DisconnectedNo TCP connection to the server. Address and IPEndPoint are reported as null in this state.
ConnectingTCP connection being established. Contributes to isTransitioning.
ConnectedTCP connection established; sets isConnected. Login may not yet have occurred.
LoggingInLogin handshake in progress (typically Connected | LoggingIn). Contributes to isTransitioning.
LoggedInAuthenticated and fully operational; sets isLoggedIn. Searches, browsing and transfers are possible only in this state.
DisconnectingConnection being torn down. Contributes to isTransitioning.

Two ServerState shapes

slskd defines two records named ServerState: the API DTO in Core/API/DTO/ServerState.cs (with Username) and the internal application-state record in Core/State.cs (which additionally exposes isConnecting and isLoggingIn and omits Username, since the username lives on the sibling User state). Both expose the same state flags and the isConnected/isLoggedIn/isTransitioning derivations. The GET /api/v0/server endpoint is the canonical read.

Connect & reconnect flow

slskd does not connect automatically unless valid credentials are configured. A dedicated ConnectionWatchdog drives (re)connection, always reading the current username/password so credential edits take effect on the next attempt without a restart:

  1. At startup, if --no-connect is set, or username/password are empty, slskd stays disconnected (it logs a warning and waits for a manual connect).
  2. Otherwise the ConnectionWatchdog starts and calls Client.ConnectAsync(address, port, username, password) with backoff + jitter on failure.
  3. On success the Connected event fires (the watchdog stops); login proceeds to LoggedIn.
  4. On disconnect, or on a LoginRejectedException, the watchdog restarts the reconnect loop.
  5. Changing address, port, username or password is marked RequiresReconnect вҖ” slskd resets the server connection to apply them.

10. Relationship to Relay mode

A single slskd instance does everything described above. Relay mode is an optional topology in which two or more slskd instances cooperate as one logical client: one controller holds the single Soulseek login and speaks all three network planes, while one or more agents connect to the controller (not to the Soulseek network) and relay their shared files into it. Agents dial out to the controller over HTTPS, so they need no inbound ports or port-forwarding of their own вҖ” useful for serving files from locations where forwarding the Soulseek listen port is impractical. Agent traffic is authenticated with an API key plus a per-agent shared secret, optionally scoped by CIDR. The controller exposes a dedicated SignalR hub at /hub/relay.

This is a deployment concern layered on top of the core architecture, not a change to the Soulseek protocol itself. See Relay Protocol for the full controller/agent handshake, message set, and security model.
