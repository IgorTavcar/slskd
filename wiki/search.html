Search

A search asks the Soulseek network "who has files matching this text?" and accumulates replies from peers over a short, bounded time window. This page documents the full search life cycle in slskd — the REST endpoints used to create, poll, stop and delete a search; the SignalR hub used to stream responses as they arrive; the option/limit knobs that govern how long a search runs and which responses are accepted; the underlying wire behaviour (FileSearch out, SearchResponse in, propagated through the distributed peer tree); and the client-side filter grammar that the slskd web UI applies to the accumulated results. Field names and defaults are grounded in slskd source and in the underlying Soulseek.NET library.

Search life cycle

A search in slskd is a persisted record with two identifiers: a Guid id chosen by the client (or generated by the server if omitted) and a numeric int32 token allocated by the Soulseek client and embedded in the wire request. The id is the handle used for all REST/SignalR operations; the token is how the server and peers correlate replies on the network.

  1. Create. The client POSTs to /api/v0/searches with a body containing searchText (and optionally id and option overrides). The controller serializes search creation through a single-permit semaphore — only one search may be initiated at a time; a concurrent request receives 429.
  2. Persist & broadcast. slskd immediately writes a record in state Requested, broadcasts CREATE over the SearchHub, and calls SoulseekClient.SearchAsync, which emits a FileSearch message to the server with the allocated token.
  3. Run. Matching peers connect back and deliver SearchResponse messages. Soulseek.NET raises a state transition to InProgress and invokes the response handler for each accepted response. slskd accumulates responses in memory and pushes rate-limited count updates (UPDATE) and per-response payloads (RESPONSE) over the hub.
  4. Complete. The search ends when its timeout elapses (measured from the last response), a limit is reached, it is cancelled, or it errors. The terminal state always carries the Completed flag, plus zero or more sub-flags (below). The final record is persisted with the accumulated responses and an endedAt timestamp; an UPDATE is broadcast with responses stripped.
  5. Retrieve / delete. The client polls GET /searches/{id} (optionally with responses) or relies on the hub stream, then optionally DELETEs the record. slskd also prunes completed searches older than a configurable retention age.
Note

The POST returns as soon as the search has been initiated — it returns the newly created record (typically in state Requested/InProgress with zero responses), not the completed result. Despite the API doc-comment "The search completed successfully", responses are gathered asynchronously on a background task; the caller must poll GET /searches/{id} or subscribe to the SearchHub to observe completion and results.

Source: src/slskd/Search/API/Controllers/SearchesController.cs, src/slskd/Search/SearchService.cs (StartAsync)

Search states

The state field is a bit-flag set ([Flags] enum Soulseek.SearchStates). A search is "done" when (state & Completed) != 0; slskd surfaces this as the derived boolean isComplete on the Search record. The reason for completion is conveyed by the additional flags OR-ed alongside Completed.

FlagValueMeaning
None0No state.
Requested1The search has been created and the FileSearch request is about to be / has been sent. Initial state written by slskd.
InProgress2The request is on the network and responses are being accepted.
Completed4The search has finished. Always present in a terminal state; combine with sub-flags below.
Cancelled8Completed because the client called PUT /searches/{id} (cancellation token tripped).
TimedOut16Completed because the inter-response timeout (searchTimeout) elapsed. This is the normal, expected completion reason.
ResponseLimitReached32Completed because the number of accepted responses reached responseLimit.
FileLimitReached64Completed because the cumulative number of accepted files reached fileLimit.
Errored128Completed abnormally (exception during execution, e.g. the client was disconnected). slskd also sets this if it must finalize a search whose normal completion was never observed.
Queued256Defined by Soulseek.NET; not used by slskd's standard network-scope search flow.

Typical terminal values seen over the API (the JSON serializes the flag set as a comma-joined string, e.g. "Completed, TimedOut"):

Serialized stateNumericInterpretation
"Completed, TimedOut"20Normal completion (timeout reached after last response).
"Completed, ResponseLimitReached"36Stopped early — enough responses.
"Completed, FileLimitReached"68Stopped early — enough files.
"Completed, Cancelled"12Stopped by the user via PUT.
"Completed, Errored"132Failed (e.g. server disconnect).
"InProgress"2Still running.
Cancellation semantics

PUT /searches/{id} trips the cancellation token only while the search is actively in progress (slskd holds the token in an in-memory dictionary keyed by id and removes it on completion). If the search has already completed, TryCancel returns false internally; the controller still returns 200 OK regardless of whether anything was actually cancelled. A search that has already finished cannot be re-stopped.

Source: Soulseek.SearchStates (Soulseek.NET src/SearchStates.cs); slskd src/slskd/Search/Types/Search.cs (IsComplete), src/slskd/Search/SearchService.cs

How a search travels the network

slskd does not query peers directly. It hands the query to the central server, which injects it into the distributed peer tree; peers that find matches open a peer connection back to the searcher and deliver results. The mechanics are fully specified on the Soulseek Wire Protocol page; the essentials as they pertain to a slskd search:

  1. Submit. slskd allocates a uint32 token (Client.GetNextToken()) and sends server message FileSearch (code 26) { token, query }. For non-network scopes the corresponding server message is used instead (see Search scope).
  2. Propagate. The server forwards the request down the distributed tree as a DistributedSearchRequest. Each participating peer matches it against its own share index.
  3. Reply. A peer with matches connects to the searcher over a peer connection and sends SearchResponse (code 9) — a zlib-compressed payload carrying username, the echoed token, the matching files[], a free-slot flag, average upload speed, queue length, and lockedFiles[].
  4. Correlate. Soulseek.NET matches the response token to the active search, applies the acceptance filters, and invokes slskd's response handler. There is no explicit "search finished" message on the wire — completion is determined locally by the inter-response timeout or a configured limit.
Note

Because results arrive over independent peer connections and completion is timeout-driven, a search is inherently best-effort: the absence of a result for a given user does not prove the user lacks the file (they may be offline, firewalled, slow to respond, or filtered out by an acceptance option). See Architecture & Concepts for the distributed-network model.

Source: src/slskd/Search/SearchService.cs (Client.SearchAsync, line ~309); message codes per soulseek-protocol.html (Soulseek.NET MessageCode)

Search scope

Soulseek supports several search scopes, modelled by Soulseek.SearchScope / SearchScopeType. Each maps to a different server message on the wire.

SearchScopeTypeValueSubjectsServer messageDescription
Network0noneFileSearch (code 26)Global search broadcast across the distributed network. The only scope slskd's REST API exposes today.
User11+ usernamesUserSearch (code 42)Search a specific user's shares.
Room2exactly 1 roomRoomSearch (code 120)Search the shares of members of a chat room.
Wishlist3noneWishlistSearch (code 103)Periodic server-driven re-search; not driven by this controller.
slskd limitation

The slskd SearchesController hard-codes SearchScope.Network: Searches.StartAsync(id, SearchQuery.FromText(request.SearchText), SearchScope.Network, …). There is no request field to select a user or room scope. The underlying ISearchService.StartAsync and Soulseek.NET both accept any scope, but it is not reachable through the public API in this revision. The DTO even carries a // todo: adjust this when additional search inputs are made available note. Clients needing user/room search must drive Soulseek.NET directly.

Source: src/slskd/Search/API/Controllers/SearchesController.cs:107; Soulseek.SearchScope, Soulseek.SearchScopeType (Soulseek.NET); server message codes per soulseek-protocol.html

Query parsing & exclusions

The searchText string is parsed by SearchQuery.FromText (Soulseek.NET). Whitespace splits it into tokens; any token beginning with - and longer than one character becomes an exclusion term (with the leading - stripped, de-duplicated), and the remaining tokens are positive terms. Exclusions are sent as part of the wire query. Separately, the removeSingleCharacterSearchTerms option (default true in Soulseek.NET; not exposed by slskd's DTO) drops one-character terms before transmission.

searchText = "aphex twin selected ambient -live -bootleg"
  terms      = ["aphex", "twin", "selected", "ambient"]
  exclusions = ["live", "bootleg"]
Note

This server-side exclusion parsing (-term) is distinct from the slskd web UI's client-side filter grammar, which also uses -term but evaluates it locally against already-received filenames. Both can be present in the same string.

Source: Soulseek.SearchQuery (Soulseek.NET src/SearchQuery.cs)

Search options & limits

The POST body may carry per-search overrides. Any omitted option falls back to the default of new Soulseek.SearchOptions() (the mapping is RequestValue ?? defaultValue in SearchRequest.ToSearchOptions). These options govern when a search completes and which responses are accepted (counted and forwarded to the handler) versus silently dropped.

Field (JSON)TypeValidationEffective defaultDescription
idguid?—server-generatedClient-chosen unique id. If supplied and already used, the request fails (DuplicateTokenException / ArgumentException → 400).
searchTextstringrequired, non-blank—The query. Must not be null, empty, or whitespace-only (otherwise 400).
searchTimeoutint32?≥ 515000Idle timeout that ends the search, measured from the last received response. Passed verbatim to Soulseek.NET, whose unit is milliseconds (default 15000 ms = 15 s). See the unit caveat below.
responseLimitint32?0 … int32 max250Maximum number of accepted responses before the search auto-completes with ResponseLimitReached.
fileLimitint32?0 … int32 max25000Maximum cumulative number of accepted files before the search auto-completes with FileLimitReached.
filterResponsesbool?—trueWhen true, the acceptance filters below (minimumResponseFileCount, maximumPeerQueueLength, minimumPeerUploadSpeed) are enforced. When false, every response is accepted.
minimumResponseFileCountint32?0 … int32 max1A response must contain at least this many files (counting both files and locked files) to be accepted.
maximumPeerQueueLengthint32?0 … int32 maxint32 maxReject responses from peers whose upload queue is longer than this. (slskd's DTO doc-comment says "1000000"; the value actually applied when the field is omitted is int.MaxValue from SearchOptions.)
minimumPeerUploadSpeedint32?0 … int32 max0Reject responses from peers whose average upload speed is below this value (bytes/sec).
Documented vs. effective defaults

The slskd SearchRequest doc-comments state defaults of fileLimit = 10000, responseLimit = 100, maximumPeerQueueLength = 1000000, and searchTimeout = 15 "in seconds". These are inaccurate for the code path: omitted fields are filled from new Soulseek.SearchOptions(), whose constructor defaults are searchTimeout = 15000, responseLimit = 250, fileLimit = 25000, maximumPeerQueueLength = int.MaxValue, minimumPeerUploadSpeed = 0, minimumResponseFileCount = 1, filterResponses = true. The "Effective default" column above reflects the actual Soulseek.NET values. The searchTimeout unit is also a known ambiguity: the slskd comment says seconds but the value is forwarded unchanged to Soulseek.NET, which treats it as milliseconds — pass 15000 (not 15) for a 15-second timeout.

Options exposed by Soulseek.NET but not mapped by slskd's DTO: removeSingleCharacterSearchTerms (default true), and the delegate hooks responseFilter/fileFilter/stateChanged/responseReceived (slskd binds the latter two internally for persistence and SignalR broadcasting; the filter delegates are available to in-process callers via ToSearchOptions(...) and the WithFilters extension, but cannot be set over REST).

Source: src/slskd/Search/API/DTO/SearchRequest.cs; Soulseek.SearchOptions (Soulseek.NET src/Options/SearchOptions.cs); src/slskd/Search/Extensions.cs

REST endpoints

All search endpoints are under /api/v0/searches and require the Any authorization policy (any authenticated principal). See Authentication & Security for how to obtain a token. When slskd is running as a Relay agent, every endpoint returns 403 Forbidden — searches are a controller-mode concern.

Create a search

POST/api/v0/searchesAny

Request body (only searchText is required; the slskd web client sends only { id, searchText }):

{
  "id": "8f2b9e4a-7c1d-4e2a-9f0b-1a2b3c4d5e6f",
  "searchText": "aphex twin selected ambient works",
  "searchTimeout": 15000,
  "responseLimit": 250,
  "fileLimit": 25000,
  "filterResponses": true,
  "minimumResponseFileCount": 1,
  "maximumPeerQueueLength": 1000000,
  "minimumPeerUploadSpeed": 0
}

Get a search

GET/api/v0/searches/{id}?includeResponses={bool}Any
ParameterInTypeDefaultDescription
idrouteguid—The search id.
includeResponsesqueryboolfalseWhen true, the responses array (with files/lockedFiles) is included; when false, it is empty.

Returns the single Search record, or 404 Not Found if no search has that id. With includeResponses=true the full response/file tree is materialized from the persisted JSON.

Get just the responses

GET/api/v0/searches/{id}/responsesAny

Returns only the responses array (always includes responses; equivalent to the responses property of GET /searches/{id}?includeResponses=true). 404 if the id is unknown.

Stop an in-progress search

PUT/api/v0/searches/{id}Any

Requests cancellation of a running search. Returns 200 OK if the search exists; 404 if the id is unknown. The search transitions to Completed, Cancelled if it was actually in progress (see the cancellation caveat). The route is documented as also producing 304 Not Modified when the search was not in progress, but the current implementation returns 200 OK unconditionally once the search is found.

Delete a search

DELETE/api/v0/searches/{id}Any

Removes the search record (and its stored responses) from the database and broadcasts DELETE over the hub. Returns 204 No Content on success, or 404 if the id is unknown. Deleting does not stop a still-running search's network activity — stop it first if it is in progress.

Source: src/slskd/Search/API/Controllers/SearchesController.cs; client wrappers in src/web/src/lib/searches.js

Real-time streaming (SearchHub)

To observe searches without polling, connect to the SignalR hub at /hub/search (negotiated under the SignalR base path; see Real-Time API & Events for connection and auth details). The hub requires the Any policy. On connect, the server invokes the client method LIST with the current set of searches (responses omitted) as a one-time snapshot; thereafter the server pushes per-event messages.

Client method SignalRConstantPayloadFired when
LISTSearchHubMethods.ListSearch[] (responses omitted)Once, on connection — current snapshot of all searches.
CREATESearchHubMethods.CreateSearch (responses null)A new search has been created (right after the record is persisted).
UPDATESearchHubMethods.UpdateSearch (responses null)A search's state or counts changed — sent on every state transition and on rate-limited (~every 250 ms) count updates while responses arrive, and on finalization.
RESPONSESearchHubMethods.Response{ "searchId": guid, "response": SearchResponse }An individual accepted response arrived. The response object is the raw Soulseek.NET SearchResponse (camelCased), letting clients render results incrementally.
DELETESearchHubMethods.DeleteSearch (responses null)A search was deleted (manually or by retention pruning).
Streaming vs. fetching

The hub never streams the accumulated responses array on UPDATE/CREATE/DELETE — those are deliberately nulled to keep socket payloads small. To get full result content, a client either consumes individual RESPONSE events as they arrive, or (after seeing UPDATE with isComplete: true) fetches GET /searches/{id}?includeResponses=true. The RESPONSE event payload uses the Soulseek.NET SearchResponse shape (same fields as below, minus slskd's added code/isLocked on files).

Source: src/slskd/Search/API/Hubs/SearchHub.cs (SearchHubMethods, SearchHubExtensions); broadcasts in src/slskd/Search/SearchService.cs

Response & file shape

A completed search's responses array consists of Response objects — one per replying peer — each carrying that peer's matching File entries split into available files and lockedFiles (files the peer restricts, e.g. to privileged users). These are slskd's projection of the Soulseek.NET SearchResponse / File; the canonical field references live on Data Model & Types.

Client-side result filter grammar

Independent of the server-side acceptance options, the slskd web UI applies a local filter to the responses it has already received. The filter is expressed as a free-text string parsed by parseFiltersFromString and applied per-file by filterResponse (in src/web/src/lib/searches.js). Any client implementing a similar result-narrowing UX can reuse this grammar; it is purely client-side and is not sent to slskd or the network.

The string is tokenized on spaces. Tokens fall into three classes: key:value numeric thresholds, bare boolean flags, and plain include / -exclude terms.

Filter tokens

TokenAliasesApplies to (File field)Effect
minbr:Nminbitrate:NbitRateDrop files with bitRate < N.
minbd:Nminbitdepth:NbitDepthDrop files with bitDepth < N.
minfs:Nminfilesize:NsizeDrop files with size < N (bytes).
minlen:Nminlength:NlengthDrop files with length < N (seconds).
minfif:Nminfilesinfolder:Nresponse fileCount + lockedFileCountDrop the entire response if it contains fewer than N files in total.
isvbr—isVariableBitRateKeep only files where isVariableBitRate === true (drops unknown/CBR).
iscbr—isVariableBitRateKeep only files where isVariableBitRate === false (drops unknown/VBR).
islossless—sampleRate, bitDepthKeep only files that have both a sampleRate and a bitDepth (heuristic for lossless formats like FLAC/WAV).
islossy—sampleRate, bitDepthKeep only files that have neither sampleRate nor bitDepth (heuristic for lossy formats like MP3).
term—filename (lowercased)Include term: every plain term must be a substring of the filename (logical AND).
-term—filename (lowercased)Exclude term: drop the file if the filename contains any excluded term.

Parsing & evaluation rules

  • Case-insensitive. Key matches use case-insensitive regexes; plain terms and filenames are lowercased before comparison.
  • Numeric values are parsed with parseInt(…, 10) against /(key):(\d+)/; a missing/zero value leaves the corresponding minimum at 0 (no effect). The matched \d+ is taken literally — minfs is bytes, not MB/KB.
  • Token classification. Any token containing : and the four reserved flag words (isvbr, iscbr, islossless, islossy) are excluded from the include/exclude term lists. Remaining tokens that start with - become exclusions (leading - stripped); all others become include terms.
  • minfif is response-level. If a response's total file count (fileCount + lockedFileCount) is below minFilesInFolder, the whole response's files are emptied before per-file filtering.
  • Per-file predicate (short-circuits to drop): a file is removed if any of: isCBR and the file is VBR or unknown; isVBR and the file is CBR or unknown; isLossless and it lacks sampleRate or bitDepth; isLossy and it has sampleRate or bitDepth; bitRate < minBitRate; bitDepth < minBitDepth; size < minFileSize; length < minLength; not all include terms are substrings of the filename; or any exclude term is a substring of the filename.
  • Both files and lockedFiles are filtered independently, and the response's fileCount/lockedFileCount are recomputed from the filtered arrays.
Comparison caveat

The numeric file fields are nullable. In the JS predicate a null compared with < a positive minimum (e.g. null < 320) evaluates to false, so a file with an unreported bitRate is not dropped by minbr — only files that report a value below the threshold are removed. This is observed behaviour of the reference implementation, not a documented guarantee.

Example combining server exclusions and client filters in one box:

aphex twin minbr:320 islossless minfif:5 -live -remaster

Source: src/web/src/lib/searches.js (parseFiltersFromString, filterResponse)

Server-side response filtering

Separately from per-search acceptance options and the UI filter, slskd can drop incoming search requests from other peers (i.e. requests slskd would otherwise answer from its own shares) using configured regular expressions. This is a privacy/abuse control, not a result filter for your own searches.

OptionCLI / EnvTypeDefaultDescription
filters.search.request--search-request-filter / SEARCH_REQUEST_FILTERstring[]emptyRegular expressions; an incoming search request matching any of them is ignored (slskd will not respond with shares).

Each entry is validated as a valid .NET regular expression at startup; an invalid pattern fails configuration validation. A related retention setting, retention.search (int?, minimum 5 minutes), controls how long completed searches are kept before being pruned and broadcast as DELETE.

Source: src/slskd/Core/Options.cs (FiltersOptions.SearchOptions, RetentionOptions.Search); src/slskd/Search/SearchService.cs (PruneAsync)

Operational notes

  • One search at a time (initiation). The single-permit limiter only guards the creation path; many searches can be in progress concurrently — the gate prevents two POSTs from racing during setup. A POST that collides returns 429.
  • Persistence. Searches are stored in a SQLite database (SearchDbContext). Responses are serialized to a single JSON column (responsesJson, not returned directly — surfaced as the responses array), so list/get-without-responses queries project the record without materializing that column for efficiency.
  • Counts vs. responses. responseCount, fileCount and lockedFileCount on the Search record are maintained live (rate-limited) during the search and reflect accepted results after server-side acceptance options. The client-side filter grammar can reduce the displayed counts further without changing these stored totals.
  • Tokens are uint32 on the wire. slskd's record exposes token as a signed int32; on the Soulseek wire it is an unsigned 32-bit integer. Treat it as an opaque correlation id.
  • Relay agents cannot search. Every endpoint and the network search itself are disabled when running in Relay-agent mode; see Relay Protocol.
