Transfers & Queueing

A transfer is a single file moving between two Soulseek peers. slskd models every download and upload as a Transfer with a flag-based state machine, persists them, and exposes them through the /api/v0/transfers REST surface. This page documents the transfer entity, the TransferStates state machine and its categories, the download enqueue/batch/query/cancel flow, the upload queue (groups, slots, priorities and queue strategies), the placement and conflict strategies, and the underlying wire-level negotiation that all of this drives.

The Transfer entity

Every download and upload is a single Transfer record. The same class serves both directions; the direction field distinguishes them. Downloads and uploads are stored and queried separately (different services and tables) but share this shape. The DTO is serialized as camelCase JSON.

Transfer state machine

state is a bit-flag enum (Soulseek.NET TransferStates). A live state is usually a composition of a coarse phase flag and a refinement flag, e.g. Queued | Locally or Completed | Succeeded. The integer values below are taken verbatim from the bit arithmetic documented in slskd's TransferStateCategories source.

FlagValueKindMeaning
None0initialNo state. Default value of a freshly constructed record before scheduling.
Requested1phaseThe transfer has been requested from the remote peer; we optimistically assume it will be allowed to enqueue.
Queued2phaseThe transfer is queued (waiting for a slot). Usually carries a Locally or Remotely refinement.
Initializing4phaseThe data connection is being established; transfer is about to start.
InProgress8phaseBytes are actively moving.
Completed16phaseTerminal phase flag. Always combined with a refinement below.
Succeeded32refineCompleted successfully. Completed | Succeeded = 48.
Cancelled64refineCompleted by user cancellation. Completed | Cancelled = 80.
TimedOut128refineCompleted because an operation timed out. Completed | TimedOut = 144.
Errored256refineCompleted with an unhandled error. Completed | Errored = 272.
Rejected512refineCompleted because the remote peer rejected the request. Completed | Rejected = 528.
Aborted1024refineCompleted because the local file could not be found / served (uploads). Completed | Aborted = 1040.
Locally2048refineQueued in our local queue (waiting for a local slot / retry backoff). Queued | Locally = 2050.
Remotely4096refineQueued in the remote peer's queue. Queued | Remotely = 4098.

Note

The serialized string is a comma-and-space joined flag list (.NET [Flags] ToString()), so order follows ascending bit value: the coarse flag first, then the refinement (e.g. "Completed, Succeeded", "Queued, Remotely"). Parse it by splitting on ", " and OR-ing the flags, or match the whole string. The web client matches whole strings — see src/web/src/lib/transfers.js.

Client logic

The web client derives three predicates from the state string (src/web/src/lib/transfers.js) — these are the canonical rules a client should reproduce:

  • Retryable: state.includes('Completed') && state !== 'Completed, Succeeded' — any failed terminal state.
  • Cancellable: state is one of InProgress, Requested, Queued, Queued, Remotely, Queued, Locally, Initializing.
  • Removable: state.includes('Completed') — any terminal state.

State categories

slskd groups state integers into hash sets (TransferStateCategories) so that database queries can use IN (...) instead of bitwise HasFlag (SQLite cannot use indexes for bitwise ops). These categories are what the API filters on — e.g. DELETE /transfers/downloads/all/completed removes every transfer whose state is in the Completed set.

CategoryMember states (integer)
SuccessfulCompleted | Succeeded (48)
FailedCancelled (80), TimedOut (144), Errored (272), Rejected (528), Aborted (1040) — each OR'd with Completed
CompletedCompleted (16) ∪ Successful ∪ Failed
QueuedRequested (1), Queued (2), Queued | Locally (2050), Queued | Remotely (4098)
InProgressInitializing (4), InProgress (8)

Place in queue & remote queueing

"Place in queue" describes how far a download is from starting in the remote peer's upload queue. slskd does not know this until it explicitly asks: GetPlaceInQueueAsync issues a wire-level place request to the peer, stores the answer in placeInQueue, and returns the integer. The value is advisory and frequently inaccurate.

For uploads, slskd is the one maintaining the queue, so it computes positions itself. The upload queue exposes two estimators (used internally when responding to a peer's place request and when forecasting):

  • EstimatePosition(user, file) — estimated position of a specific queued file. For RoundRobin groups it interleaves every user's queue (worst-/average-case interleaving); for FirstInFirstOut groups it is the file's index in the group's time-of-enqueue order (including in-progress uploads), which is accurate.
  • ForecastPosition(user) — position a user would land at if they enqueued now: 0 if a slot is free; for RoundRobin, (users in group) + 1; for FIFO, (total uploads in group) + 1.

Source: src/slskd/Transfers/Downloads/DownloadService.cs (GetPlaceInQueueAsync) · src/slskd/Transfers/Uploads/UploadQueue.cs

Download flow

To download from a peer, a client enqueues one or more files for a username. slskd:

  1. Resolves the user's IP endpoint and rejects the request if the user is blacklisted or appears offline (UserOfflineException → 404).
  2. Inserts a Transfer record per file in state Queued | Locally (deduplicating against transfers that are already queued or in progress — those land in the failed bucket).
  3. Requests the file from the peer over the wire; on acceptance the state moves to Queued | Remotely (or directly toward Initializing if a slot is immediately available).
  4. Streams bytes to an incomplete file, then on success moves the file to its final destination and sets Completed | Succeeded.
  5. On failure, retries up to the configured attempt limit with exponential-ish backoff (downloads only), resuming or overwriting the partial file per RetryPartialStrategy.
Concurrency limit

Both download-enqueue endpoints share a process-wide semaphore of 1 concurrent operation (a SemaphoreSlim(2,2) entered with Wait(0) per request — only one in-flight at a time effectively, because each request holds it for the duration). A second concurrent enqueue returns 429 "Only one concurrent operation is permitted. Wait until the previous request completes".

Download batches

A Batch is a lightweight grouping of related download transfers, optionally tied to the search that produced them and optionally overriding the destination directory. The batch is the recommended way to enqueue (the single-user endpoint is obsolete). A batch is purely a tracking/association construct: the application carries no aggregate counters on it (the source explicitly defers that).

Upload flow & the queue

slskd serves uploads in response to remote peers' enqueue requests. There is no API to "start" an upload — they are demand-driven. When a peer requests a file:

  1. slskd validates the request against the requesting user's group limits (queued/daily/weekly file, megabyte, and failure caps from Limits options) using UserUploadStatistics. If a limit is exceeded the request is rejected.
  2. A Transfer is created in state Queued | Locally and an Upload is enqueued into the in-memory UploadQueue.
  3. The transfer awaits a slot. When the peer's data connection is ready, the queue marks the upload ready (upload.ready = now).
  4. The queue's Process() loop selects the next ready upload according to group priority and strategy, marks it started, consumes a slot, and releases the TaskCompletionSource so the transfer proceeds to InProgress.
  5. On completion the slot is returned to the group it was drawn from.

The Upload queue record (in-memory, not the DTO):

FieldTypeDescription
UsernamestringRequesting user.
FilenamestringRequested file.
GroupstringGroup the slot was pinned to (set when started).
EnqueuedDateTimeWhen enqueued (FIFO ordering key).
ReadyDateTime?When the peer connection became ready (RoundRobin ordering key); null until ready.
StartedDateTime?When a slot was granted; null until started.
Global cap

The queue refuses to release any upload once the sum of used slots across all groups reaches the global upload-slots count, even if a group has spare slots. This deliberately keeps queue ordering under slskd's control rather than letting requests stack behind Soulseek.NET's internal upload semaphore.

Source: src/slskd/Transfers/Uploads/{UploadQueue,UploadService,UploadGovernor}.cs · src/slskd/Transfers/Types/{Upload,UserUploadStatistics}.cs

Upload groups, slots & priorities

Every remote user maps to exactly one upload group. Groups have a priority, a slot count, and a queue strategy. The queue processes groups in ascending priority order (lower number = higher priority) and stops after releasing the first ready upload it finds.

Three groups are built in; user-defined groups are added on top:

GroupPrioritySlotsStrategyMembership
privileged0 (hard-coded, highest)= global upload-slotsFirstInFirstOut (fixed)Soulseek-privileged users.
defaultconfig (≥ 1; default 1)min(group.slots, global)config (default roundRobin)Users not privileged, not leechers, in no other group.
leechersconfig (≥ 1; default 1)min(group.slots, global)config (default roundRobin)Users below the shared files/directories thresholds.
user-definedconfig (≥ 1)min(group.slots, global)configExplicit member lists.

The UploadGroup record (also the shape of GET .../uploads group info internally):

FieldTypeDescription
namestringGroup name.
priorityint32Lower is processed first.
slotsint32Maximum concurrent uploads for the group.
usedSlotsint32Slots currently in use.
slotAvailableboolComputed: usedSlots < slots.
strategyQueueStrategyRoundRobin or FirstInFirstOut.

Strategy semantics (selection within a group):

  • RoundRobin (0) — among ready uploads, the one ready longest wins (orders by Ready). Spreads slots across users; a faster downloader naturally advances faster.
  • FirstInFirstOut (1) — among ready uploads, the one enqueued earliest wins (orders by Enqueued). Privileged users are fixed to FIFO to mimic the official client.

Strategies & placement

Four enums govern how downloaded files are placed on disk and how partial files are handled. All four are serialized/configured by their lowercased name (e.g. "resume", "overwrite", "rename", "roundrobin", "firstinfirstout") via the options system.

EnumMember (value)Meaning
RetryPartialStrategyOverwrite (0)On retry, discard the partial file and start over from offset 0.
Resume (1) (default)On retry, resume using the partial file's current size as the start offset.
DestinationExistsStrategyRename (0) (default)If the destination file already exists, write under a renamed (appended) name.
Overwrite (1)Overwrite the existing destination file.
DestinationConflictStrategyRename (0)Resolve a name conflict by appending to the new file's name.
Overwrite (1)Overwrite the conflicting file.
QueueStrategyRoundRobin (0) (group default)Prioritize uploads by the time they became ready.
FirstInFirstOut (1)Prioritize uploads by the time they were enqueued.

Destination placement

The destination subdirectory for a download is derived from a token-templated pattern (Transfers.Download.Destination.Subdirectory, default ${SOURCE_DIRECTORY}). A batch's options.destination overrides this pattern. The pattern is matched against ${TOKEN} placeholders; tokens include the (sanitized) source username, source directory, batch external id, and search text. All resolved values are sanitized and forced to a relative, non-traversing path; an absolute or traversing result throws SlskdException as a backstop.

Filename sanitization

There is no sanitizeFilenames on/off option in the current source. Sanitization is unconditional: every path segment that reaches disk is passed through FileSafety.SanitizePath/SanitizePathSegment/GetFileNameSafely, and the API rejects any request whose filename contains a path-traversal segment up front (400).

Per-group limits

Each group's upload.limits (and the global upload.limits) define queued, daily and weekly caps, each with optional files (int32?), megabytes (int32?) and failures (int32?) — all null (unlimited) by default. These are evaluated against the requesting user's UserUploadStatistics when an incoming enqueue request arrives; exceeding any limit rejects the request.

Wire-level negotiation

Everything above sits on top of the native Soulseek peer-to-peer transfer handshake. A client speaking the raw protocol (rather than driving slskd) performs the following. Message codes are peer-message codes on the P (peer) connection; see Soulseek Wire Protocol for the framing.

Code 43 QueueUpload

The downloader asks the uploader to enqueue a file. Payload: the filename string. The uploader responds either by accepting (eventually sending a TransferRequest when a slot frees) or by sending an UploadDenied/QueueFailed reason.

Code 40 TransferRequest

Sent by the side that will send the file (the uploader for a normal download). Payload: a direction (0 = download / 1 = upload from the sender's perspective), a token (uint32), the filename, and the fileSize (uint64). The token keys the subsequent file connection.

Code 41 TransferResponse

Reply to a TransferRequest. Payload: the token, an allowed flag, and either the fileSize (when allowed) or a rejection message (e.g. "Queued", "File not shared."). A "Queued" response maps to the Queued | Remotely state.

Code 44 PlaceInQueueRequest · Code 44 PlaceInQueueResponse

The downloader requests its current place; the uploader replies with the filename and an integer place. This is exactly what GET /downloads/{username}/{id}/position triggers, and what the upload queue's position estimator answers when slskd is the uploader.

File connection ('F')

Once a transfer is allowed, the receiving side opens (or pierces a firewall to) a dedicated file connection keyed by the negotiated token. The first bytes the downloader sends on this connection are the 4-byte token, followed by an 8-byte (uint64) file offset indicating where to begin streaming — this is the resume mechanism. For a fresh download the offset is 0; for a Resume retry it is the size of the on-disk partial file (slskd's internal startOffset). The uploader then streams raw file bytes from that offset until fileSize bytes have been delivered.
