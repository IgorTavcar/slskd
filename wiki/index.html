Overview

This is a reference-grade, language-agnostic specification of slskd and the Soulseek peer-to-peer file-sharing network it participates in. It is written for engineers building a client, application, integration, or service in any language that must interoperate with the Soulseek platform — either by speaking the native Soulseek wire protocol directly, or by driving a running slskd daemon over its HTTP REST and SignalR APIs. The specification is derived from the slskd source tree and the embedded Soulseek.NET library (v10.0.1), cross-checked against the community-maintained protocol documentation.

What slskd is

slskd is a modern, self-hostable, client-server application for the Soulseek file-sharing network. It runs as a long-lived daemon (or Docker container) on a server in your network or in the cloud, connects to the central Soulseek server and to peers, shares files, performs searches, and transfers files — and exposes all of this functionality through a browser-based web UI, an HTTP REST API, and a SignalR real-time API.

Internally, slskd embeds the Soulseek.NET library to implement the native Soulseek wire protocol. Soulseek.NET handles the framed, little-endian, length-prefixed binary message exchange with the central server and with peers; slskd layers configuration, persistence (SQLite), share management, queueing, an integration/event system, and the public web/API surface on top. The two layers are conceptually distinct and are documented separately in this spec.

Note

The HTTP/HTTPS ports (5030/5031) carry the slskd web/API surface and are unrelated to the Soulseek protocol. The Soulseek listen port (50300) is where slskd accepts incoming connections from other Soulseek peers. A client that speaks the wire protocol does not use the REST/SignalR ports at all, and vice versa.

Two interoperation surfaces

This specification is organized around a fundamental distinction. There are two completely separate ways to interoperate with the Soulseek platform, and which one you target determines which pages of this spec apply to you.

Surface A

Speak the native wire protocol

Connect directly to the central Soulseek server and to peers, exchanging framed binary messages. This is the lowest-level, fully independent path: you implement the protocol yourself and behave as a first-class Soulseek client. No slskd instance is required.

 Surface B

Drive a slskd daemon over REST/SignalR

Run (or connect to) a slskd instance and control it over its HTTP REST API and SignalR real-time hubs. slskd handles all wire-protocol details; you issue JSON commands and consume JSON events. This is the fastest path to integration in any language with an HTTP client.

Conventions used throughout

These conventions hold across the entire specification. Page-specific exceptions are called out where they occur.

Wire protocol (Surface A)

The Soulseek wire protocol is a binary, little-endian, length-prefixed message protocol over TCP.

ElementEncoding
Integer (32-bit)int32, little-endian, typically unsigned for codes/lengths
Integer (64-bit)int64, little-endian (e.g. file sizes, byte offsets)
Boolean1 byte: 0x00 = false, 0x01 = true
Stringint32 length prefix (byte count) followed by that many UTF-8 bytes; no NUL terminator
Message frame4-byte LE total length, then 4-byte LE message code, then the payload. The length field counts the code plus payload (it does not count its own 4 bytes)
Warn

Message codes overlap between connection types: a code value is only meaningful relative to its connection context (server vs. peer vs. distributed). The same numeric code can mean different things on different connections. Always interpret a code together with the connection it arrived on. See Soulseek Wire Protocol.

Authentication (Surface B)

The slskd API supports two credential types, evaluated by separate authentication schemes:

  • JWT bearer token — obtained by logging in with the configured web username/password; presented as an Authorization: Bearer <token> header.
  • API key — a configured static key, presented in a request header; keys carry a role (e.g. readonly, readwrite) and may be scoped to an IP/CIDR.

Endpoints are protected by role-based policies (Any, ReadOnly, ReadWrite, Guest, Administrator). The default web credentials are slskd / slskd and must be changed for any exposed instance. See Authentication & Security for the full model, and note that this is entirely separate from the Soulseek network login (Surface A), which uses your Soulseek account credentials.

Source: src/slskd/Program.cs (AddAuthentication, JwtBearer, ApiKey policies), README.md

Glossary

Terms used throughout the specification. Where a term has both a network meaning and a slskd-specific meaning, both are noted.

TermDefinition
Soulseek serverThe central, authoritative Soulseek server. Clients connect to it to log in, search, look up users, and join rooms. It coordinates the network but does not relay file data — files move directly between peers.
PeerAnother Soulseek client on the network. Peers connect to each other directly (subject to firewalls/NAT) to browse shares, exchange user info, request files, and transfer them.
Peer connection (P)A direct peer-to-peer connection used for peer-type messages (browse, user info, file/queue requests). Conventionally denoted connection type P.
Distributed network (D)An overlay tree of peer connections used to propagate search queries across the network without flooding the central server. Conventionally denoted connection type D.
Branch root / branch levelIn the distributed search tree, the branch root is the peer at the top of a client's branch; the branch level is that client's depth in the tree. The server and parent peers advertise/relay these so searches fan out efficiently.
File connection (F)A dedicated peer connection opened to transfer the bytes of a single file. Separate from the peer (P) connection used for control messages. Conventionally denoted connection type F.
Listen portThe TCP port a client advertises and listens on for incoming peer/distributed connections. slskd defaults to 50300. Reachability typically requires port forwarding.
Token(1) Wire: a client-generated int32 correlation id attached to a request (search, transfer, connection) so responses can be matched to it. (2) slskd API: a JWT bearer token authenticating an API caller. Context disambiguates.
Search tokenThe token (correlation id) assigned to a specific search, used to associate incoming search responses with the originating query.
TickerA short text message ("ticker"/topic) associated with a chat room, set by members and broadcast to the room.
Slot / upload slotA unit of upload concurrency. A client offers a limited number of simultaneous upload slots; when none are free, incoming download requests are queued. "Has free slot" is part of a client's advertised state.
Queue / place in queueWhen an upload slot is unavailable, a requested file is placed in the uploader's queue. The downloader can request its place in queue — its 1-based position pending transfer.
BrowseRequesting and receiving the full (or partial) directory/file listing of another user's shares over a peer connection.
ShareA directory tree a client makes available to the network. slskd scans configured share roots into an index that answers searches and browse requests.
Locked filesFiles that appear in a user's share listing/search results but are restricted from download (e.g. reserved for privileged or specific users). Browse and search responses can carry a separate locked set.
Watch / watched userSubscribing to a user so the server pushes status/stat updates for them. slskd watches users it interacts with to track online/offline status, slots, and stats.
Privileged userA user with Soulseek privileges (historically purchased to support the network), granting priority placement in others' upload queues.
WishlistA saved/standing search that is re-issued periodically so the client is notified when matching files later appear on the network.
Relay controllerIn relay mode, the slskd instance that connects to the Soulseek network and acts as the client. Agents connect to it. See Relay Protocol.
Relay agentIn relay mode, a slskd instance that connects to a controller (not to Soulseek) and relays files from its local storage to the controller. Think of it as remote storage for the controller.
DaemonThe long-running slskd process/service that maintains the Soulseek connection and serves the web/API surface.
slskd instanceOne running deployment of the slskd daemon, with its own configuration, data directory, and (in non-relay or controller mode) its own Soulseek login.
