Shares & Browsing

A Soulseek peer offers files to the network by sharing local directories. slskd scans configured directories into a SQLite cache, advertises aggregate counts to the server, answers peer browse and directory-contents requests with a backslash-delimited virtual directory tree, and resolves the virtual paths that other peers request back to physical files for upload. This page specifies the share configuration syntax, the share/host/scan model, how shares are scanned, cached and advertised, the browse protocol in both directions, the REST /api/v0/shares surface, and virtual-to-physical path resolution.

Concepts & terminology

Three forms of a path coexist in the share subsystem. Keeping them distinct is essential to interoperation.

TermExampleMeaning
Local path/home/me/Music/test/foo.txtThe physical file on the host's filesystem. Native OS separators. Never sent over the network.
Alias / remote path rootMusicThe single-segment name under which a shared directory is presented to peers. Defaults to the share folder's name.
Masked / remote / virtual pathMusic\test\foo.txtThe path as seen by remote peers and as it travels on the wire: the local path with its share root replaced by the alias and all separators normalized to backslash (\).
Note

Soulseek paths use the Windows backslash \ as the directory separator regardless of the host OS. slskd normalizes every masked path with path.Replace('/', '\\') before storing it in the cache or sending it on the wire. Clients consuming or producing masked paths must treat \ as the separator and must not URL-style or POSIX-style normalize them.

Source: src/slskd/Common/CommonExtensions.cs (NormalizePathForSoulseek), src/slskd/Shares/ShareScanner.cs.

Share configuration syntax

Shares are configured through shares.directories (CLI -s/--shared, env SLSKD_SHARED_DIR), an array of strings. Each entry is a single path with optional decorators. The full grammar of one entry:

[ ('-' | '!') ]  [ '[' alias ']' ]  <absolute-local-path>
DecoratorSyntaxEffect
Exclusionleading - or !The path (and everything beneath it) is excluded. Used to carve a subdirectory out of a broader share.
Alias[Name] prefixPresents the share under Name instead of the folder's own name. Improves privacy when the local path leaks a username.
(none)bare pathAlias defaults to the last path segment, e.g. D:\Music → alias Music.

Examples

shares:
  directories:
    - 'D:\Music'                         # shared as "Music"
    - '!D:\Music\Personal Recordings'    # excluded subdirectory of the above
    - '[Library]\\users\John Doe\Music'  # aliased as "Library" to hide the username

Path & alias rules (validated at startup)

RuleDetail
Must be absoluteMust begin with /, X: (drive letter), or \\ (UNC). Relative paths (~/dir, ../dir) are rejected.
No root mountSharing / on a unix-like OS is explicitly rejected.
No traversal segmentsEntries containing /./, /../ (or backslash equivalents) are rejected.
Alias non-emptyAn alias may not be null, empty, or whitespace; an entry like [Music] with no path is rejected ("does not specify a path").
Alias no separatorsAliases may not contain / or \ (the alias is a single directory segment).
Aliases uniqueTwo shares may not resolve to the same alias ("collide"). Specify an alias to disambiguate.
Paths uniqueThe same physical path may not be shared twice under different aliases.

Trailing separators are trimmed (share.TrimEnd('/', '\\')) and duplicate entries are de-duplicated before processing. Shares are processed longest-local-path-first, so a subdirectory share can be aliased independently of its parent.

Source: src/slskd/Core/Options.cs (SharesOptions.ValidateShares), src/slskd/Shares/Types/Share.cs, src/slskd/Shares/ShareService.cs (Configure), docs/config.md.

Share filters (masks)

shares.filters (CLI --share-filter, env SLSKD_SHARE_FILTER) is an array of regular expressions. During a scan, any directory or file whose physical path matches at least one filter is excluded from the cache. Filters are case-insensitive by default; case sensitivity follows the global flags.case_sensitive_regex option. The default list is empty.

shares:
  filters:
    - \.ini$
    - Thumbs.db$
    - \.DS_Store$
Note

Filters are matched against the original (local) filename, not the masked path, and an invalid regular expression fails option validation. This is distinct from search filters (filters.search), which reject incoming search requests rather than files.

Source: src/slskd/Shares/ShareScanner.cs, src/slskd/Core/Options.cs (SharesOptions.ValidateFilters).

Share, Host, Scan & StorageMode

Share

Each configured directory becomes one Share. Serialized to JSON in camelCase by the REST API.

FieldTypeDescription
idstringSHA-1 hash of the remotePath (= the alias). Stable identifier used in /shares/{id} routes.
aliasstringThe alias; equals the bracketed name or the last path segment.
isExcludedbooltrue if the raw entry began with - or !.
localPathstringThe physical directory path (OS separators).
rawstringThe original configuration string, decorators included.
remotePathstringThe path root presented to peers. Currently equal to alias.
directoriesint?Directory count under this share, populated after a scan; null until then.
filesint?File count under this share, populated after a scan; null until then.

Source: src/slskd/Shares/Types/Share.cs. See also data-model.html#share.

Host

A Host groups the shares that originate from one instance. The local instance is always present as a host named local (Program.LocalHostName); additional hosts are relay agents, each named after the agent's configured InstanceName.

FieldTypeDescription
namestringHost name; "local" for the local instance, otherwise the agent's instance name.
isLocalHostbooltrue when name == "local".
sharesShare[]The shares hosted by this host.

The browse and search responses sent to the network are the union of all hosts' shares — peers cannot tell that files came from different physical machines.

Source: src/slskd/Shares/Types/Host.cs, src/slskd/Shares/ShareService.cs.

Scan

Each scan writes a Scan record into the share database. It records the option set in effect, so the application can detect on the next start whether the configuration changed and a re-scan is required.

FieldTypeDescription
startedAtlongUnix epoch (milliseconds) when the scan began.
optionsJsonstringThe serialized shares options at scan time. Compared against current options on init.
endedAtlong?Unix epoch (ms) at completion; null if the scan did not finish.
suspectboolMarked true when a re-scan has been requested; forces a fresh scan on init.

Source: src/slskd/Shares/Types/Scan.cs, src/slskd/Shares/ShareService.cs (InitializeAsync).

StorageMode

The share cache is a SQLite database. shares.cache.storage_mode (CLI --share-cache-storage-mode, env SLSKD_SHARE_CACHE_STORAGE_MODE) selects where it lives. The option string is parsed into this enum.

ValueNumericBehavior
memory0 (default)SQLite in-memory database. Faster lookups, lower CPU/disk, higher RAM. On startup the cache is restored from an on-disk backup; if the backup is missing/invalid/stale, a full re-scan runs.
disk1On-disk SQLite file. Lower memory, slower lookups, more I/O. On startup the on-disk file is validated; if invalid, it is restored from backup, otherwise a full re-scan runs.

Changing storage_mode requires a restart ([RequiresRestart]). After every successful scan the cache is also backed up to a secondary database file regardless of mode.

Source: src/slskd/Shares/Types/StorageMode.cs, src/slskd/Core/Options.cs (ShareCacheOptions), src/slskd/Shares/ShareService.cs.

Scanning & the share cache

A scan enumerates every shared directory and file, reads audio/video metadata, and writes masked-path records into the SQLite cache. Scanning is I/O-intensive; shares.cache.workers (default Environment.ProcessorCount, range 1–128) controls how many files are processed concurrently.

Scan algorithm

  1. Compile the filters regex list (case-insensitive unless case_sensitive_regex is set).
  2. Insert a new Scan record with the serialized options.
  3. For each non-excluded share, recursively enumerate subdirectories (skipping Hidden and System attributes, ignoring inaccessible paths), discarding directories matched by a filter; then add the share roots themselves.
  4. Remove directories that fall under any excluded share (a directory is excluded if its physical path starts with an excluded share's local path).
  5. For each surviving directory, insert a masked directory row, then enumerate its files (non-recursive, skipping hidden/system). Each file matched by a filter is counted as filtered and skipped; the rest are read into a Soulseek.File and inserted as masked file rows.
  6. On completion, set endedAt, back up the cache database, and recompute per-share directory/file statistics.

Metadata extraction

For files with a recognized audio or video extension, slskd uses TagLib to populate file attributes. The attribute set and ordering deliberately mimic established clients:

Codec classAttributes emitted (in order)
Lossy (BitsPerSample = 0)BitRate, Length (seconds), VariableBitRate (0/1)
Lossless (BitsPerSample > 0)Length (seconds), SampleRate, BitDepth

Recognized audio extensions include aa aax aac aiff ape dsf flac m4a m4b m4p mp3 mpc mpp ogg oga wav wma wv webm; video includes mkv ogv avi wmv asf mp4 m4p m4v mpg mpe mpv m2v. Files of any other extension are shared with no attributes. Metadata read failures are logged and the file is still shared (attributes omitted).

Source: src/slskd/Shares/ShareScanner.cs, src/slskd/Shares/SoulseekFileFactory.cs.

Path masking & virtual paths

Masking is the deterministic transform applied during the scan that converts a physical path into the path peers see. For a file under share S:

masked = original.ReplaceFirst(S.LocalPath, S.RemotePath).Replace('/', '\\')

Worked example (from the relay walkthrough), with share /home/kubuntu/Music aliased Music:

Local pathMasked / wire path
/home/kubuntu/Music/test/foo.txtMusic\test\foo.txt

The masked path is the SQLite primary key (maskedFilename) and the only path form transmitted on the wire. The reverse mapping (masked → local) is what file resolution performs when serving a download.

Source: src/slskd/Shares/ShareScanner.cs, src/slskd/Shares/SqliteShareRepository.cs.

Advertising shares to the network

slskd advertises shares to the Soulseek server in two ways, which interoperating clients must replicate.

Aggregate counts (server message)

After login, after each scan, and whenever the share host set changes, slskd reports its total directory and file counts to the server. In Soulseek.NET this is SetSharedCountsAsync(directories, files), which emits the server message SharedFoldersFiles.

Code 35 SharedFoldersFiles Client → Server

Reports the number of shared folders and files. Payload: uint32 folder count, uint32 file count. These counts feed the per-user "files / folders" display other clients show, and the leecher detection thresholds that gate uploads.

Note

The message code 35 is the Soulseek server protocol code as implemented by Soulseek.NET; slskd does not construct the raw message itself. Verify the wire layout against soulseek-protocol.html.

Source: src/slskd/Application.cs (Client.SetSharedCountsAsync).

Browse-response warming

The full directory tree a peer receives when browsing is not built on demand for every request. After a scan (and on host add/remove), slskd builds the complete BrowseResponse once, serializes it to bytes, and writes it to browse.cache in the data directory. Incoming browse requests stream this pre-warmed file (as a RawBrowseResponse), avoiding per-request CPU. If the cache file is missing when a request arrives, it is rebuilt on the fly.

Source: src/slskd/Application.cs (CacheBrowseResponse, BrowseResponseResolver).

Locked / private shares

The Soulseek browse protocol distinguishes ordinary directories from locked (private) directories. slskd's browse response is built entirely from the public directory list; it does not populate the locked-directory section of BrowseResponse. Blacklisted requesters receive an empty response instead (see below). (Inferred from the absence of any locked-directory population in the resolver.)

Browse & directory listing (incoming)

Two peer-protocol requests let a remote user inspect your shares. slskd answers both from its share cache.

Browse request (whole tree)

When a peer browses you, Soulseek.NET invokes slskd's browse resolver. The response is the warmed browse.cache file. Behavior:

RequesterResponse
Allowed userThe full pre-warmed directory tree (union of all hosts' shares).
Blacklisted user / IPAn empty BrowseResponse (no directories).

The tree is constructed by BrowseAsync: every known directory is primed with an empty Directory entry (so directories containing only subdirectories still appear), then files are grouped by their normalized parent directory and merged in. Each file is sent with just its filename (not the full masked path) to save bandwidth.

Directory-contents request (one directory)

A peer can also request the contents of a single directory by its masked path. slskd's directory resolver:

  • Rejects paths containing traversal segments (. / ..) — returns an empty list and logs a warning.
  • Returns a single empty Directory for blacklisted requesters.
  • Otherwise returns one Directory populated with the files in that masked directory (looked up across all hosts), filenames only.

Source: src/slskd/Application.cs (BrowseResponseResolver, DirectoryContentsResponseResolver), src/slskd/Shares/ShareService.cs (BrowseAsync, ListDirectoryAsync).

Browsing another user (outgoing)

To browse a remote user's shares, drive the slskd Users API. slskd issues the peer browse request through Soulseek.NET and returns the assembled directory tree.

GET/api/v0/users/{username}/browseAny

Retrieves the full directory tree shared by {username}. Returns an array of Directory (HTTP 200). Returns 404 if the user is offline (UserOfflineException) or blacklisted. Forbidden (403) when the instance is running as a relay agent.

GET/api/v0/users/{username}/browse/statusAny

Returns the progress of an in-flight browse for {username} as a decimal (HTTP 200), or 404 if no browse is tracked. Browse progress is tracked while the response transfers and the tracker entry is removed ~5 seconds after the browse completes. Forbidden (403) on a relay agent.

Browse response JSON shape

Both /users/{username}/browse and the local /shares/contents endpoints return the same shape: an array of directories, each with its masked name and a list of files. Field names are camelCase.

[
  {
    "name": "Music\\test",
    "files": [
      {
        "code": 1,
        "filename": "foo.txt",
        "size": 1024,
        "extension": "txt",
        "attributes": []
      }
    ]
  },
  {
    "name": "Music\\Albums\\Example",
    "files": [
      {
        "code": 1,
        "filename": "01 - Track.flac",
        "size": 31457280,
        "extension": "flac",
        "attributes": [
          { "type": 1, "value": 220 },
          { "type": 4, "value": 44100 },
          { "type": 5, "value": 16 }
        ]
      }
    ]
  }
]
Note

In a browse response the filename is just the leaf name; the containing directory is given by the parent Directory.name. In search results the filename is the full masked path. The attributes[].type codes are the FileAttributeType enum (1 = BitRate, 2 = Length, 3 = VariableBitRate, 4 = SampleRate, 5 = BitDepth — verify against data-model.html).

Source: src/slskd/Users/API/Controllers/UsersController.cs, src/web/src/lib/shares.js. See data-model.html#directory and data-model.html#file.

Shares REST API

The SharesController exposes the local share configuration, lets you browse your own shares, and controls scans. All endpoints require the Any policy.

GET/api/v0/sharesAny

Lists all shares grouped by host. The response is an object keyed by host name, each value an array of Share objects.

{
  "local": [
    {
      "id": "0f6e2c…",
      "alias": "Music",
      "isExcluded": false,
      "localPath": "/home/me/Music",
      "raw": "/home/me/Music",
      "remotePath": "Music",
      "directories": 42,
      "files": 1337
    }
  ],
  "example_agent": [ /* shares relayed from an agent host */ ]
}
GET/api/v0/shares/{id}Any

Returns the single Share whose id matches (searched across all hosts). 404 if not found.

GET/api/v0/shares/contentsAny

Returns the entire local share tree as an array of Directory — the same structure a peer receives when browsing you. (lib/shares.js: browseAll().)

GET/api/v0/shares/{id}/contentsAny

Returns the contents of one share as an array of Directory, scoped to that share's remotePath prefix. 404 if the share id is unknown. (lib/shares.js: browse({ id }).)

PUT/api/v0/sharesAny

Initiates a scan of the configured shares (fire-and-forget). Returns 200/204 on acceptance, or 409 Conflict (ShareScanInProgressException) if a scan is already running. (lib/shares.js: rescan().)

DELETE/api/v0/sharesAny

Cancels a running scan. Returns 204 if a scan was cancelled, or 404 if none was in progress. (lib/shares.js: cancel().)

Method & pathReturnsNotes
GET /api/v0/shares{ host: Share[] }Shares grouped by host.
GET /api/v0/shares/{id}Share404 if unknown.
GET /api/v0/shares/contentsDirectory[]Whole local tree.
GET /api/v0/shares/{id}/contentsDirectory[]One share's tree. 404 if unknown.
PUT /api/v0/shares200/204Start scan. 409 if already scanning.
DELETE /api/v0/shares204Cancel scan. 404 if not scanning.

Source: src/slskd/Shares/API/Controllers/SharesController.cs, src/web/src/lib/shares.js.

File resolution (download path)

When a peer requests a file (in a download/queue request), slskd receives the masked path and must map it back to a physical file to upload. This is ResolveFileAsync(remoteFilename).

  1. Look up remoteFilename against the local host's repository: SELECT originalFilename, size FROM files WHERE maskedFilename = @maskedFilename. The match is an exact equality on the masked path (no prefix/LIKE matching).
  2. If found locally, return (host = "local", localFilename, size).
  3. If not found locally, iterate relay agent hosts one by one, querying each host's repository, and return the first match as (agentName, localFilename, size).
  4. If no host resolves the path, throw NotFoundException ("could not be resolved to a physical file").

Logged resolution from the relay walkthrough:

[INF] Resolved Music\test\foo.txt to physical file /home/kubuntu/Music/test/foo.txt on host 'example_agent'
Security

Because resolution is an exact masked-path equality against the scanned cache, a peer cannot request an arbitrary local path: only files that were actually scanned and masked are resolvable, and only their recorded originalFilename is opened. The directory-contents resolver additionally rejects traversal segments. Clients implementing the native protocol should apply the same constraints.

Source: src/slskd/Shares/ShareService.cs (ResolveFileAsync), src/slskd/Shares/SqliteShareRepository.cs (FindFileInfo), docs/relay.md.

Relay hosts & remote share databases

In a relay topology, an agent instance holds the physical files while a controller instance is connected to the Soulseek network. The agent scans its own shares locally, then uploads its entire SQLite share database to the controller. The controller registers the agent as an additional Host, merges the agent's directories and files into its browse response, and re-warms browse.cache.

Controller-side log sequence when an agent connects:

[INF] Beginning download of shares to /tmp/slskd/share_example_….db
[INF] Download of shares from example_agent complete (48.0 KB)
[INF] Loading shares from agent example_agent
[INF] Warming browse response cache...
[INF] Shares from agent example_agent ready.
[INF] Browse response cached successfully in 25ms

From this point, browse, search and directory-contents responses include the agent's files transparently, and file resolution for an agent-hosted path returns that agent's host name; the controller then requests the physical file stream from the agent over the relay channel (see relay.html).

Note

Browse and directory endpoints under /users/{username}/... are forbidden (403) when slskd runs as a relay agent — only the controller speaks to the network.

Source: docs/relay.md, src/slskd/Shares/ShareService.cs (AddOrUpdateHost, ResolveFileAsync), src/slskd/Users/API/Controllers/UsersController.cs.
