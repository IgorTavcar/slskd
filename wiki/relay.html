Relay Protocol

The Relay lets two or more slskd instances act as a single Soulseek client. One instance — the Controller — connects to the Soulseek network and presents itself as the client. One or more Agents connect outward to the controller (no inbound ports at the agent) and act as remote file storage: they upload their share databases to the controller, and stream file content on demand. This page specifies the transport, the challenge/response handshake, the SignalR hub and REST surface, the credential algorithm, and each relay flow as an ordered sequence, for engineers building a compatible agent or controller in any language.

Concept & Topology

A relay configuration consists of exactly one Controller and one or more Agents. The controller is a normal slskd instance that logs in to the Soulseek server and serves the network. Agents do not connect to Soulseek; they connect only to the controller and relay files to it. Conceptually, agents are "remote storage" for the controller.

Agents always initiate the connection (agent → controller), so the agent's network needs no port forwarding, inbound firewall rules, or public address. The only trust the agent places outward is in the controller's address. All controller-bound traffic is authenticated and, when the controller is configured for HTTPS, end-to-end encrypted.

        ┌──────────────────────────────┐      ┌────────────────────────────────┐

A remote Soulseek peer browsing or downloading from the controller's username transparently receives files that physically reside on an agent; the controller proxies the byte stream. The remote peer is unaware that a relay exists. See architecture.html for where the Relay sits in the overall topology and shares.html for how agent shares merge into the controller's browse response.

Transport & Authentication

An agent connects to the controller over two channels, both targeting the same base address (relay.controller.address, e.g. https://controller.example.com:5031):

  • SignalR hub at /hub/relay — a persistent bidirectional connection used for the handshake, control messages, and request signalling.
  • REST endpoints under /api/v0/relay/controller/... — used to stream bulk payloads (the share database, file content) that do not fit the hub's message-oriented model.

Two independent secrets gate the connection:

SecretCarried asValidated byPurpose
API key (relay.controller.api_key)SignalR access token (bearer), and the X-API-Key header on REST callsThe controller's web authentication layer (web.authentication.api_keys)Coarse gate: identifies the caller as an authorized API client with a sufficient role. Required to open the hub or call any REST endpoint.
Per-agent shared secret (relay.controller.secret on the agent; relay.agents.<key>.secret on the controller)A computed credential (never the secret itself) in the hub Login call and the X-Relay-Credential headerThe Relay service, per agent, via challenge/responseFine gate: proves the caller is the specific named agent and possesses that agent's secret. Bound to each request token to prevent replay.
Note

The shared secret is never transmitted. It is used locally to derive an encryption key; only the encrypted form of a controller-issued token (the "credential") crosses the wire. Possession of the API key alone does not let a caller impersonate an agent or download arbitrary files.

The agent presents the API key as the SignalR bearer access token, configured via AccessTokenProvider on the hub connection. On REST calls it is passed in the X-API-Key header. The API key must map (on the controller) to an entry under web.authentication.api_keys with a sufficient role and an IP/CIDR scope that permits the agent's address. See authentication.html for API key configuration and roles.

Source: src/slskd/Relay/RelayClient.cs, src/slskd/Program.cs (MapHub<RelayHub>("/hub/relay"))

Credential Algorithm

Every authenticated relay operation is bound to a single-use, controller-issued token. The agent answers a token by producing a credential: the token encrypted under a key derived from the shared secret. The controller independently recomputes the expected credential and compares. The algorithm is identical for the login challenge, share-upload tokens, file-stream tokens, and download tokens — only the token value differs.

  1. Derive a 48-byte key with PBKDF2 / HMAC-SHA256, 1000 iterations:
    key = PBKDF2(password = sharedSecret, salt = UTF8(agentInstanceName), prf = HMAC-SHA256, iterations = 1000, length = 48)
  2. Split the 48 bytes into a 32-byte AES-256 key and a 16-byte IV (CBC, 128-bit block).
  3. Encrypt the UTF-8 bytes of the token string: cipher = AES-256-CBC(UTF8(token), key, iv).
  4. Encode the ciphertext as Base62. The resulting string is the credential.
Important

The PBKDF2 salt is the agent's instance name (a non-secret identifier), and the IV is derived deterministically from the key material — there is no random nonce. This makes the credential a deterministic function of (secret, instanceName, token), which is exactly what lets the controller recompute and compare it. Security rests on the secrecy of sharedSecret and the single-use, short-lived nature of each token, not on IV randomness. A compatible agent must reproduce this exact construction.

The challenge token is a random 32-byte value, Base62-encoded, cached against the connection id for only 10 seconds. Request tokens (share, file, download) are GUIDs cached for their respective windows (see the flow tables). Tokens are validated once and (except download tokens, which permit retries) removed from the cache, preventing replay.

Source: src/slskd/Relay/RelayClient.cs (ComputeCredential), src/slskd/Relay/RelayService.cs (TryValidateAuthenticationCredential), src/slskd/Common/Cryptography/{Pbkdf2,Aes,Random}.cs

Connection & Handshake Sequence

The handshake begins the instant the SignalR connection is established. The controller drives it by pushing a challenge; the agent answers with Login.

  1. Agent connects. The agent opens the SignalR connection to /hub/relay, presenting the API key as the bearer access token. If the controller's relay is disabled or it is not in Controller/Debug mode, the connection is aborted.
  2. Controller challenges. On connect, the controller generates a random challenge token, caches it against the connection id (10 s TTL), and invokes the client method Challenge(token).
    Controller log: Agent connection <id> from <ip> established. Sending authentication challenge <token>...
  3. Agent responds. The agent computes the credential from the challenge token (see Credential Algorithm) and invokes the server method Login(agent, challengeResponse), where agent is its instance name.
    Agent log: Relay controller sent an authentication challenge → Logging in...
  4. Controller validates, in order: (a) the supplied agent name matches a configured agent; (b) the remote IP falls within that agent's cidr; (c) the credential equals the controller's recomputed expected value. Any failure throws UnauthorizedAccessException and the agent is deregistered. On success the controller registers the Agent (name + IP) against the connection id.
    Controller log: Agent connection <id> from <ip> authenticated as agent <name>
  5. Agent uploads shares. Once logged in, the agent proceeds to the share-upload flow (below).
Warn

On reconnect (SignalR automatic reconnect, or a manual restart) the controller issues a new challenge and the agent must re-authenticate and re-upload shares from scratch. The challenge token's 10-second TTL means a compatible agent must answer promptly; a slow responder will fail validation and be retried by the connection loop.

Source: src/slskd/Relay/API/Hubs/RelayHub.cs (OnConnectedAsync, Login), src/slskd/Relay/RelayClient.cs (HandleAuthenticationChallenge)

RelayHub Methods

The hub is /hub/relay, decorated [Authorize] (a valid API key is required to connect). Method names below are the exact SignalR invocation names — a compatible agent must register handlers and invoke targets using these literal names. The hub is strongly typed: server→client calls are defined by the IRelayHub interface.

Note

Hub invocations are signalling only. The controller cannot push a large payload over the hub; instead it sends a token via the hub and the agent pulls/pushes the bytes over a separate authenticated HTTP request. This keeps file streams off the hub's message channel and lets them be processed in-flight without buffering.

RelayController Endpoints

All routes are under api/v{version}/relay (API version 0 → /api/v0/relay/...). The agent endpoints control the local agent and are JWT-authenticated (operator-facing); the controller/... endpoints receive agent traffic and are API-key-authenticated. Every endpoint also enforces that the relay is enabled and the instance is in a compatible mode, returning 403 Forbidden otherwise.

PUT/api/v0/relay/agentAdministrator

Starts the local relay client and connects to the configured controller. Valid only when this instance is in Agent/Debug mode. JWT policy. Returns 200 OK. This is what the web UI's "connect" button calls.

DELETE/api/v0/relay/agentAdministrator

Stops the local relay client and disconnects from the controller. Agent/Debug mode only. JWT policy. Returns 204 No Content.

POST/api/v0/relay/controller/shares/{token}ReadWrite Administrator

Agent → controller. Uploads the agent's share database. Controller/Debug mode only; API-key policy, role ReadWrite or Administrator. multipart/form-data body with a JSON shares field and a database file part. Headers X-Relay-Agent (instance name) and X-Relay-Credential required. Request size limit: 1 TiB.

POST/api/v0/relay/controller/files/{token}ReadWrite Administrator

Agent → controller. Streams a requested file's content. Controller/Debug mode only; API-key policy, role ReadWrite or Administrator. multipart/form-data with a single file part; the stream is read in-flight (no buffering) and forwarded straight to the waiting Soulseek upload. Headers X-Relay-Agent and X-Relay-Credential required. Request size limit: 10 TiB. The {token} matches the id from RequestFileUpload.

GET/api/v0/relay/controller/downloads/{token}Any

Agent → controller. Retrieves a completed download file the controller advertised via NotifyFileDownloadCompleted. Controller/Debug mode only; API-key policy, role Any. Headers X-Relay-Agent, X-Relay-Credential, and X-Relay-Filename-Base64 (the requested filename, Base64-encoded) required. Returns the file as application/octet-stream. Unlike the other endpoints the token is not consumed on validation, to permit retries.

The agent endpoints require an Administrator role under the JWT policy — they are operator controls on the agent instance, not part of agent↔controller traffic. The brief's expectation of a ReadWrite role applies to the controller/shares and controller/files upload endpoints, which is what an agent actually calls. The controller/downloads endpoint accepts any role because its real gate is the per-file relay credential.

Flow: Share Upload

After login, the agent publishes its shares so the controller can advertise them to the Soulseek network. This also happens on every reconnect, and whenever the agent's local shares change (SynchronizeAsync).

  1. Agent backs up its share database to a temporary file and serializes its list of shares to JSON.
  2. Agent invokes hub BeginShareUpload() → controller generates a GUID token, caches it against the agent name (5-minute TTL), and returns it.
  3. Agent POSTs to /api/v0/relay/controller/shares/{token} a multipart/form-data body containing a shares field (JSON) and a database file part (the SQLite share DB), with headers X-API-Key, X-Relay-Agent, and X-Relay-Credential (credential = encrypted token).
  4. Controller validates the credential against the cached share token, writes the DB to a temp file, validates it, then loads it as a share host named after the agent and warms the browse cache.
  5. Controller responds 200 OK; the agent logs "Shares uploaded. Ready to relay files."
Note

The uploaded shares JSON is the same Share shape used elsewhere in slskd (see shares.html and data-model.html). The controller merges each agent's host into its aggregate browse response, so remote peers see a unified directory tree.

Source: src/slskd/Relay/RelayClient.cs (UploadSharesAsync), RelayController.cs (UploadShares), RelayService.cs (GenerateShareUploadToken, HandleShareUploadAsync)

Flow: File Relay

When a remote Soulseek peer downloads a file that physically lives on an agent, the controller proxies the byte stream from the agent through to the peer without ever staging the whole file. The upload service resolves the virtual path to (host, physicalPath); host is the agent name.

File-info request (pre-flight)

Before transferring, the controller confirms the file exists on the agent and learns its length:

  1. Controller invokes hub RequestFileInfo(filename, id) on the resolved agent and awaits a response (default 3 s timeout).
  2. Agent resolves the virtual filename to a local path, stats it, and invokes ReturnFileInfo(id, exists, length).
  3. Controller uses the result to validate the share against disk before starting the transfer.
Note

The file-info round-trip can be skipped if the controller is configured with OptimisticRelayFileInfo (it then trusts the uploaded share metadata instead of querying the agent). Default: off.

File content relay

  1. Controller resolves the virtual path to an agent + physical file.
    Log: Resolved <virtual> to physical file <path> on host '<agent>'
  2. Controller caches a file token (a GUID id) against the agent name (TTL = the request timeout), then invokes hub RequestFileUpload(filename, startOffset, id) and waits for the stream.
  3. Agent opens the local file, seeks to startOffset, and POSTs it to /api/v0/relay/controller/files/{id} as multipart/form-data, with headers X-API-Key, X-Relay-Agent, X-Relay-Credential.
  4. Controller validates the credential against the cached file token, captures the request body stream in-flight, and feeds it directly into the Soulseek upload to the remote peer. The HTTP request stays open for the entire transfer.
  5. When the Soulseek upload completes, the controller completes the HTTP request (200 OK); the agent logs upload success.
  6. If the agent cannot serve the file (missing/locked), it invokes hub NotifyFileUploadFailed(id, exception) so the controller fails fast instead of waiting for timeout.
Tip

The startOffset parameter means relayed uploads support resume: a peer reconnecting mid-transfer causes the agent to seek and stream from the requested byte, exactly like a non-relayed upload.

Source: src/slskd/Relay/RelayService.cs (GetFileInfoAsync, GetFileStreamAsync, HandleFileStreamResponseAsync), RelayClient.cs (HandleFileInfoRequest, HandleFileUploadRequest), RelayController.cs (UploadFile), src/slskd/Transfers/Uploads/UploadService.cs

Flow: Download Relay

Optionally, completed downloads (files the controller fetched from the Soulseek network on the user's behalf) can be pushed back to agents, so the files land on home storage rather than the cloud controller. Enabled per-agent with relay.controller.downloads = true.

  1. A download completes on the controller. For each registered agent the controller caches a download token (GUID id) against the agent name (10-minute TTL) and invokes hub NotifyFileDownloadCompleted(filename, id), where filename is relative to the controller's downloads directory.
  2. An agent with downloads = true issues a GET to /api/v0/relay/controller/downloads/{id} with headers X-API-Key, X-Relay-Agent, X-Relay-Credential, and X-Relay-Filename-Base64 (the filename, Base64-encoded).
  3. Controller validates the credential against the cached download token and the supplied filename, then streams the file as application/octet-stream. The token is preserved (not consumed) to permit retries.
  4. Agent writes the stream to its local downloads directory, localizing the path for its OS (handling Windows↔Linux path separators). The agent retries up to 3 times on failure.
Note

Agents with downloads = false silently ignore NotifyFileDownloadCompleted and never call the endpoint. The controller still notifies all agents; suppression happens client-side.

Source: src/slskd/Relay/RelayService.cs (NotifyFileDownloadCompleteAsync), RelayClient.cs (HandleNotifyFileDownloadCompleted), RelayController.cs (DownloadFile)

Flow: Agent Lifecycle

The agent's connection is managed automatically by the relay client, but can also be controlled through the operator API.

EventTriggerEffect
Register / connectPUT /api/v0/relay/agent, or automatic on startup when relay.enabled & mode agentStarts the hub connection retry loop (indefinite, exponential backoff 1–30 s), runs the handshake, uploads shares.
Deregister / disconnectDELETE /api/v0/relay/agentStops the hub connection; state → Disconnected.
Drop & reconnectSignalR automatic reconnect (interval policy 0,1,3,10,30,60 s)On reconnect the agent re-runs login and re-uploads shares; controller deregisters the old connection record on disconnect.

Security Recap

  • Agents initiate everything. No inbound ports, firewall holes, or public address are needed at the agent. The agent trusts only the controller's URL.
  • Two-factor gating. Every controller-bound request requires both a valid API key (coarse, role/CIDR-scoped at web.authentication.api_keys) and a per-agent credential derived from the shared secret (fine, per-agent, replay-resistant).
  • The shared secret never leaves the agent. Only encrypted, single-use, short-lived token credentials cross the wire.
  • CIDR scoping at two layers. The agent's cidr restricts which source IPs may authenticate as that agent (checked at Login), and the API key's cidr independently restricts where the key may be used. Even if both secrets leak, an attacker outside the CIDR ranges is rejected.
  • HTTPS recommended. When the controller forces HTTPS, all hub and REST traffic — including the API key and credentials — is encrypted in transit.
  • Path-traversal defense in depth. Filenames in every relay endpoint are rejected if they contain traversal segments, and physical paths are resolved with safe-combine helpers.

For API key creation, roles, and JWT auth used by the operator-facing agent endpoints, see authentication.html. For the hub transport conventions shared with other realtime features, see realtime-api.html.
