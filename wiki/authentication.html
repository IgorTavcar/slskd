Authentication & Security

slskd protects its REST API and SignalR hubs with ASP.NET Core authentication. A client authenticates with one of two credential schemes — a JWT bearer token obtained by logging in with a username and password, or a pre-configured API key sent in a header. Each credential carries a role that gates which endpoints it can reach. This page documents the login flow, the two schemes, the role model, how the same credential authenticates SignalR connections, the "passthrough" mode used when authentication is disabled, and the security implications of CIDR scoping and reverse-proxy deployment.

Overview & schemes

slskd exposes two endpoint families — a versioned REST API rooted at /api/v0 and a set of SignalR hubs rooted at /hub. Both are protected by the same authentication subsystem, configured once at startup in Program.cs. A caller presents one credential; the framework resolves it to a principal carrying a name and a single role, and the endpoint's authorization policy decides whether to admit the request.

SchemeCredentialTransportHow obtained
JWT bearerSigned JSON Web Token (HS256)Authorization: Bearer <token> header (REST); ?access_token=<token> query parameter (SignalR)POST /api/v0/session with username + password
API keyPre-shared static string (16–255 chars)X-API-Key: <key> header (REST); see SignalR for hubsConfigured under web.authentication.api_keys (or the primary key via -k / SLSKD_API_KEY)

The scheme names registered with ASP.NET Core are Bearer (the JWT default, JwtBearerDefaults.AuthenticationScheme), ApiKey, and — only when authentication is disabled — Passthrough.

Source: src/slskd/Program.cs (ConfigureAspDotNetServices, ~L822–960); src/slskd/Common/Authentication/ApiKeyAuthentication.cs; src/web/src/lib/api.js

Note

Authentication is enabled by default. The default username and password are both slskd; the documentation strongly recommends changing both before exposing the instance. Authentication can be globally disabled with web.authentication.disabled (CLI -X/--no-auth, env SLSKD_NO_AUTH) — see Passthrough.

Authentication policies

Every protected endpoint is decorated with [Authorize(Policy = …)]. Three policies are defined in AuthPolicy; each names a set of acceptable authentication schemes and requires an authenticated user.

Policy constantStringAccepted schemes (auth enabled)Typical use
AuthPolicy.Any"Any"ApiKey and BearerThe vast majority of endpoints and all hubs. Accepts either credential type.
AuthPolicy.JwtOnly"Jwt"Bearer onlyState-mutating administrative actions (e.g. updating options, restart/shutdown). An API key cannot satisfy this policy on the REST path.
AuthPolicy.ApiKeyOnly"ApiKey"ApiKey onlyDefined but reserved; little direct use in controllers.

Because Any lists the ApiKey scheme first, an X-API-Key header is evaluated before the bearer token. A request to an Any endpoint succeeds if either scheme produces a valid authenticated principal whose role also satisfies any Roles = constraint on the action (see Roles).

Warn

JwtOnly endpoints reject API-key credentials on the REST path: the policy only admits the Bearer scheme. To call an administrative endpoint such as PATCH /api/v0/options you must hold a JWT (i.e. log in), not an API key — and that JWT's role must be Administrator.

Source: src/slskd/Common/Authentication/AuthPolicy.cs; src/slskd/Program.cs (AddAuthorization, ~L846–866)

Roles & authorization

Every credential resolves to exactly one Role. The enum is defined in slskd.Authentication.Role:

RoleValueBadgeMeaning
ReadOnly0ReadOnlyRead-only access. Default role for configured API keys.
ReadWrite1ReadWriteRead plus mutating operations (queue downloads, send chat, etc.) that are not administrator-gated.
Guest2GuestDefined in the enum and referenced by the GuestOrAdministrator grouping; reserved for limited access scenarios.
Administrator3AdministratorFull access, including administrator-only endpoints. Default for the JWT issued by login and for the primary API key.

Authorization is enforced two ways. The policy selects which scheme(s) may authenticate; an optional Roles = argument on [Authorize] restricts which roles are admitted. The role groupings used in the source are defined in AuthRole:

AuthRole constantRole list (comma-separated)Admits
AnyReadOnly,ReadWrite,Guest,AdministratorAny authenticated role
AdministratorOnlyAdministratorAdministrator
ReadWriteOrAdministratorReadWrite,AdministratorReadWrite Administrator
GuestOrAdministratorGuest,AdministratorGuest Administrator

The role is carried in the principal as the ASP.NET Core role claim (ClaimTypes.Role, serialized in the JWT as http://schemas.microsoft.com/ws/2008/06/identity/claims/role) with the role name as its value (e.g. "Administrator"). When no Roles = is specified, only the policy (scheme) is enforced and any authenticated role is admitted.

Note

Most endpoints use [Authorize(Policy = AuthPolicy.Any)] with no role constraint — they admit any authenticated principal. Administrative actions add Roles = AuthRole.AdministratorOnly and frequently also restrict the policy to JwtOnly. For example, OptionsController and ApplicationController gate option-update, restart, shutdown, and GC operations with [Authorize(Policy = AuthPolicy.JwtOnly, Roles = AuthRole.AdministratorOnly)]. See REST API Reference for the per-endpoint matrix.

Source: src/slskd/Common/Authentication/Role.cs, AuthRole.cs; src/slskd/Core/Security/SecurityService.cs (GenerateJwt); src/slskd/Core/API/Controllers/OptionsController.cs, ApplicationController.cs

JWT login flow

A JWT is obtained by POSTing credentials to the session endpoint. The current implementation authenticates only the single configured administrator account (web.authentication.username / password); a successful login always yields an Administrator token.

POST/api/v0/sessionAnonymous

Request body — LoginRequest (JSON, camelCase):

FieldTypeDescription
usernamestringMust equal web.authentication.username.
passwordstringMust equal web.authentication.password.
 
POST /api/v0/session HTTP/1.1
Content-Type: application/json

{ "username": "slskd", "password": "slskd" }

Responses

StatusCondition
200Credentials valid. Body is a TokenResponse.
400Body missing, or username/password missing/blank.
401Credentials did not match.
403Returned (Forbid) when the daemon runs in headless mode — the web UI/login is disabled.

Response body — TokenResponse (JSON, camelCase):

FieldTypeDescription
tokenstringThe serialized, signed JWT. Send this as the bearer token.
tokenTypestringAlways "Bearer" (JwtBearerDefaults.AuthenticationScheme).
expiresint64Expiry as a Unix time in seconds (token exp/ValidTo).
issuedint64Issued-at as a Unix time in seconds (ValidFrom).
notBeforeint64Not-before (nbf) as a Unix time in seconds.
namestringValue of the token's Name claim (the username).
 
HTTP/1.1 200 OK
Content-Type: application/json

{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ...",
  "tokenType": "Bearer",
  "expires": 1750636800,
  "issued": 1750032000,
  "notBefore": 1750032000,
  "name": "slskd"
}

Token contents

The token is signed with HMAC-SHA256 using web.authentication.jwt.key as the symmetric key. Claims emitted by SecurityService.GenerateJwt:

ClaimValueNotes
ClaimTypes.Name (…/claims/name)usernameThe authenticated identity.
nameusernameConvenience duplicate of the name.
ClaimTypes.NameIdentifierrandom GUIDUnique per token.
ClaimTypes.Role (…/claims/role)role name, e.g. AdministratorDrives Roles = authorization.
iatissued-at, Unix secondsSet explicitly as a string claim.
issslskd (Program.AppName)Validated — see below.
nbf / expUnix secondsValidity window.

Validation parameters applied by the bearer handler: issuer must equal slskd (ValidateIssuer = true); signature is required and validated against the configured key; lifetime is validated with a 5-minute clock skew; audience is not validated. The token TTL defaults to 604800000 ms (7 days) and is configurable via web.authentication.jwt.ttl (milliseconds; minimum 3600).

Warn

If jwt.key is left unset, a fresh random 32-byte key is generated at every start. This is secure but invalidates all previously issued tokens on restart, forcing re-login. Supply a stable key (≥32 chars) if you need tokens to survive restarts. Anyone who holds the key can mint valid tokens — treat it as a secret.

Source: src/slskd/Core/Security/SecurityService.cs (GenerateJwt); src/slskd/Core/API/DTO/TokenResponse.cs, LoginRequest.cs; src/slskd/Program.cs (AddJwtBearer, ~L868–882); src/slskd/Core/Options.cs (JwtOptions, ~L2216–2238)

Session endpoints

The SessionController provides three routes used to authenticate and to probe the auth configuration.

POST/api/v0/sessionAnonymous

Log in. See JWT login flow.

GET/api/v0/sessionAny

Validate the presented credential ("whoami" no-op). Returns 200 if the bearer token or API key is valid, 401 otherwise. The web UI calls this on load to detect an expired token. Policy: AuthPolicy.Any — both a JWT and an API key are accepted. The body is empty on success.

GET/api/v0/session/enabledAnonymous

Returns a bare JSON boolean: true when authentication is enabled, false when disabled. This is the negation of web.authentication.disabled as read at startup. Decorated [AllowAnonymous], so it is callable without any credential and is the canonical way for a client to discover whether it must authenticate.

GET /api/v0/session/enabled HTTP/1.1

HTTP/1.1 200 OK
Content-Type: application/json

true

Source: src/slskd/Core/API/Controllers/SessionController.cs

API keys

API keys are static strings configured ahead of time that let a non-browser client authenticate without performing a login. Each key carries a name (its config dictionary key), a role, and an optional CIDR allow-list. Keys are sent in the X-API-Key request header for REST calls.

GET /api/v0/searches HTTP/1.1
X-API-Key: 0123456789abcdef0123456789abcdef

Configuration

Keys are defined under web.authentication.api_keys as a map of name → key options:

FieldTypeDefaultDescription
keystring—The secret value. Must be 16–255 characters.
rolestringReadOnlyOne of ReadOnly, ReadWrite, Guest, Administrator (case-insensitive). Validated against the Role enum.
cidrstring0.0.0.0/0,::/0Comma-separated CIDR allow-list (IPv4 and/or IPv6). The default matches any address. IPv4-mapped IPv6 (::ffff:…) CIDRs are rejected at validation.
 
web:
  authentication:
    api_keys:
      my_api_key:
        key: <some string between 16 and 255 characters>
        role: readwrite
        cidr: 0.0.0.0/0,::/0

Primary API key

A single "primary" key may also be supplied on the command line (-k/--api-key) or via the SLSKD_API_KEY environment variable. Because those sources are plain strings, the role and CIDR are encoded as semicolon-separated tuples and prepended to the key value:

role=ReadOnly;cidr=0.0.0.0/0,::/0;<16-to-255-character-key>

If only a bare key is supplied (no tuples), the primary key defaults to role Administrator and CIDR 0.0.0.0/0,::/0 — note this differs from the ReadOnly default of named keys. The primary key is checked before the named-keys map during authentication. A random suitable value can be generated with --generate-secret.

Validation behavior

On each request the handler reads X-API-Key; if absent it returns "no result" (allowing another scheme to try). If present, SecurityService.AuthenticateWithApiKey:

  1. Matches the supplied string exactly against the primary key (if any) and the named keys; an unmatched key raises a not-found error and the request is rejected (401).
  2. Normalizes an IPv4-mapped IPv6 caller address (::ffff:…) back to IPv4.
  3. Confirms the caller IP falls within at least one CIDR of the matched key; otherwise an out-of-range error is raised and the request is rejected.
  4. On success, builds a principal with the key name as the identity and the key's role.
Danger

API keys do not expire. Anyone who captures a key has indefinite access at that key's role. Using API-key (or any) authentication over plain HTTP exposes the key to eavesdroppers. Prefer HTTPS, scope keys with CIDR, and use a distinct key per consumer so a single key can be revoked without disrupting others.

Source: src/slskd/Common/Authentication/ApiKeyAuthentication.cs; src/slskd/Core/Security/SecurityService.cs (AuthenticateWithApiKey, ctor); src/slskd/Core/Options.cs (ApiKeyOptions, ~L2243–2294; WebAuthenticationOptions.Validate)

SignalR authentication

The SignalR hubs are mounted at the following paths and each requires authentication:

Hub pathAuthorizationPurpose
/hub/application[Authorize(Policy = AuthPolicy.Any)]Application state, server connection, transfer events.
/hub/logs[Authorize(Policy = AuthPolicy.Any)]Live log stream.
/hub/search[Authorize(Policy = AuthPolicy.Any)]Live search responses.
/hub/relay[Authorize]Relay agent connections — see Relay Protocol.

Browser SignalR clients cannot set arbitrary headers on the WebSocket handshake, so the token is passed as the access_token query-string parameter. A JwtBearerEvents.OnMessageReceived handler detects requests whose path begins with /hub and resolves the token in this order:

  1. If ?access_token=… is present, it is used as the token (takes precedence over any header).
  2. Else, if an Authorization: Bearer … header is present, the bearer value is extracted. The handler then attempts to interpret that value as an API key: if it validates (including CIDR), a fresh short-lived JWT (TTL 1000 ms) is minted for that key's name and role and used as the connection token. If it is not a valid API key, the raw value is passed through as a (presumably) JWT for the normal bearer pipeline to validate.
wss://host:5031/hub/application?access_token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Note

For an HTTP-header SignalR client you may present an API key as Authorization: Bearer <api-key>; the /hub handler converts a valid key into a short-lived JWT internally. For browser clients, put a JWT (obtained from login) in ?access_token=. The short 1000 ms TTL of the key-derived token only needs to cover the handshake; the established connection persists independently of that token's expiry.

Source: src/slskd/Program.cs (JwtBearerEvents.OnMessageReceived, ~L884–926; MapHub, ~L1113–1116); src/slskd/Core/API/Hubs/ApplicationHub.cs, LogsHub.cs; src/slskd/Search/API/Hubs/SearchHub.cs; src/slskd/Relay/API/Hubs/RelayHub.cs

Disabled / passthrough authentication

When web.authentication.disabled is true (CLI -X/--no-auth, env SLSKD_NO_AUTH), slskd registers a single Passthrough authentication scheme in place of bearer/API-key. The PassthroughAuthenticationHandler unconditionally returns a successful ticket for the username Anonymous with role Administrator. All three policies (Any, JwtOnly, ApiKeyOnly) are wired to this scheme, so every request — including administrator-only endpoints — is admitted without any credential.

The daemon logs "Authentication of web requests is DISABLED" at startup in this mode. On the client side, the web UI detects disabled auth via GET /session/enabled and engages "passthrough": it stores the sentinel token value "n/a" and stops attaching an Authorization header to requests.

Danger

Disabling authentication grants full Administrator control to anyone who can reach the API — there is no credential check at all. Only do this on a trusted, isolated network, or behind a reverse proxy that performs its own authentication. Never expose a passthrough instance to the public internet.

Source: src/slskd/Common/Authentication/PassthroughAuthentication.cs; src/slskd/Program.cs (disabled branch, ~L930–960); src/web/src/lib/session.js, token.js; src/web/src/config.js (tokenPassthroughValue = 'n/a')

CIDR scoping & reverse proxies

CIDR scoping (API keys only) is evaluated against the socket remote address of the connection — HttpContext.Connection.RemoteIpAddress. This has important consequences behind any ingress device.

Warn

Behind a reverse proxy, ingress controller, or load balancer, the remote IP seen by slskd is the proxy's address, not the original client's. slskd deliberately does not honor X-Forwarded-For (or similar) headers, because a malicious client could forge them. CIDR filtering therefore does not distinguish real clients in a proxied deployment — enforce IP restrictions at the point of ingress instead.

URL base. When slskd runs under a sub-path, set web.url_base (default /; e.g. /slskd, env SLSKD_URL_BASE). All API and hub paths are then prefixed with that base. A client must build request URLs against the same base: the web client computes {base}/api/v0 and {base}/hub from the configured urlBase. slskd advertises the base in the X-URL-Base response header (CORS-exposed).

CORS. The server registers a permissive AllowAll CORS policy (any origin, any header, any method, credentials allowed) and exposes the X-URL-Base and X-Total-Count headers. This permissiveness means the network boundary — not CORS — is the security perimeter.

HTTPS. HTTPS is enabled by default on port 5031 (option web.https); plain HTTP listens on port 5030. web.https.force redirects HTTP to HTTPS. Because both JWTs and API keys travel in headers (and tokens travel in the SignalR query string), transport encryption is the only thing preventing credential capture on the wire.

Source: src/slskd/Program.cs (AddCors, ~L824–829); src/slskd/Core/Options.cs (UrlBase ~L2056, HttpsOptions ~L2300–2372); src/web/src/config.js; docs/config.md (Authentication, Remote Configuration); docs/reverse_proxy.md

Metrics & relay authentication

Two adjacent surfaces use different credential mechanisms and are worth calling out.

Prometheus metrics

When metrics.enabled is true, the metrics endpoint (default URL /metrics) is protected by HTTP Basic auth using metrics.authentication.username / password — independent of the JWT/API-key system. It can be opened to anonymous access via metrics.authentication.disabled (which logs a warning). Send Authorization: Basic <base64(user:pass)>; a mismatch returns 401.

Source: src/slskd/Program.cs (metrics endpoint, ~L1121–1160); src/slskd/Core/Options.cs (Metrics.Authentication)

Relay

The relay subsystem (controller ↔ agent) layers an additional shared secret on top of API-key authentication. An agent connects to the controller's /hub/relay using an API key (for which the controller must have at least one key configured), and additionally proves possession of a per-agent secret. The relay hub is gated by a bare [Authorize] (any authenticated principal) and then performs its own secret check. Forcing HTTPS for relay traffic is strongly recommended so that API keys and agent secrets are not exposed. Full details are on the Relay Protocol page.

Source: src/slskd/Relay/API/Hubs/RelayHub.cs; docs/config.md (Remote Configuration / Relay)

How to authenticate (recipes)

0. Discover whether auth is required

GET /api/v0/session/enabled
→ 200  true     # must authenticate
→ 200  false    # passthrough: send no credential; you are Administrator

1. REST with a JWT (username/password)

# Log in
POST /api/v0/session
Content-Type: application/json

{ "username": "slskd", "password": "slskd" }

→ 200
{ "token": "eyJ…", "tokenType": "Bearer", "expires": 1750636800, ... }

# Use the token on subsequent requests
GET /api/v0/searches
Authorization: Bearer eyJ…

# Re-validate / detect expiry at any time
GET /api/v0/session
Authorization: Bearer eyJ…
→ 200 (valid)  |  401 (expired/invalid → log in again)

2. REST with an API key

GET /api/v0/transfers/downloads
X-API-Key: 0123456789abcdef0123456789abcdef
→ 200   # role of the key must satisfy the endpoint
        # JwtOnly endpoints (e.g. PATCH /api/v0/options) reject API keys

3. SignalR (browser-style, query parameter)

# Obtain a JWT via POST /api/v0/session, then:
wss://host:5031/hub/application?access_token=eyJ…
# (negotiate + websocket; the access_token applies to both)

4. SignalR (header-capable client, API key)

# A non-browser client may present an API key as a bearer value;
# the /hub handler converts a valid key into a short-lived JWT.
GET /hub/application/negotiate
Authorization: Bearer 0123456789abcdef0123456789abcdef
Tip

A robust client should: (1) call GET /session/enabled once; if false, send no credential. (2) Otherwise prefer an API key for headless/long-running integrations (no expiry handling) or log in for a JWT when administrator-only endpoints are needed. (3) On any 401 from a JWT, re-login; API keys never expire but may be revoked (also surfacing as 401).
