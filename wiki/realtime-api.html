Real-Time API & Events

slskd exposes live application state through four Microsoft SignalR hubs (/hub/application, /hub/logs, /hub/search, /hub/relay) and a separate, durable events log persisted to the database and surfaced under /api/v0/events. SignalR is the push channel for interactive UIs; the events log is the integration channel that drives outbound webhooks and shell scripts. This page enumerates every server→client method, client→server invocation, and payload shape for each hub, documents the event catalog and the EventRecord store, and gives a language-agnostic recipe for connecting from a non-.NET client.

Two channels: SignalR vs. Events

slskd offers two distinct real-time mechanisms. They overlap in subject matter but differ fundamentally in delivery guarantees and intended consumers.

AspectSignalR hubsEvents log
TransportPersistent connection (WebSocket / SSE / long-poll) via SignalR hub protocolPlain REST polling, plus optional push to webhooks/scripts
DurabilityEphemeral — delivered only to clients connected at the momentDurable — persisted to a SQLite table (EventRecord), survives restarts
DeliveryBest-effort broadcast to all connected clientsAt-least-once dispatch to integrations; replay via paging
GranularityFine: every state mutation, every log line, every search responseCoarse: a small set of "something happened" milestones (file complete, message received, …)
Use it forLive UI: progress bars, log tails, search result streaming, connection statusIntegrations: notifications, post-processing, automation, audit trail
AuthJWT (query string) or API key (header); policy Any for app/logs/searchJWT or API key; policy Any
Rule of thumb

If a missed message would corrupt your view, you want the durable events log (poll on reconnect to backfill). If you are rendering a live screen and only the latest value matters, use a SignalR hub. The two are independent subsystems; a build/post-process pipeline should never depend on SignalR delivery.

SignalR basics for slskd

The four hubs are mapped in Program.cs after the routing/auth middleware:

endpoints.MapHub<ApplicationHub>("/hub/application");
endpoints.MapHub<LogsHub>("/hub/logs");
endpoints.MapHub<SearchHub>("/hub/search");
endpoints.MapHub<RelayHub>("/hub/relay");

These are Microsoft ASP.NET Core SignalR hubs, not raw WebSockets. A client must speak the SignalR hub protocol: a POST {hub}/negotiate?negotiateVersion=1 handshake that returns a connectionId / connectionToken and the list of available transports, followed by a transport connection (WebSockets preferred, then Server-Sent Events, then long polling) and a per-connection JSON handshake frame terminated by the ASCII record separator 0x1E.

Authentication

All four hubs are decorated with [Authorize]. The application/logs/search hubs use policy Any (a valid JWT or API key with any role); the relay hub uses the default authenticated policy and an additional challenge/response handshake (see RelayHub and Relay Protocol).

Because browsers cannot set an Authorization header on a WebSocket, slskd accepts the token via the access_token query-string parameter for any path beginning with /hub. The middleware checks the query string first; if absent, it falls back to a Bearer Authorization header (which may carry a JWT or an API key — an API key is internally exchanged for a short-lived JWT).

// JWT in the query string (browser style)
wss://host:5030/hub/application?access_token=eyJhbGciOiJIUzI1NiIs...

// or an Authorization header on the negotiate request (non-browser client)
Authorization: Bearer <jwt-or-api-key>
Note

The query-string token takes precedence over the Authorization header. SignalR sends the token on every transport request, including the long-poll/SSE fallbacks, so it must be a query parameter for those transports too. See Authentication & Security for token issuance and API-key configuration.

Source: src/slskd/Program.cs (OnMessageReceived, MapHub, AddSignalR); src/web/src/lib/hubFactory.js; src/web/src/config.js

A note on method-name casing

The server defines its server→client method names as UPPER-CASE constants (e.g. STATE, OPTIONS, LOG, BUFFER, LIST, CREATE). The SignalR JSON hub protocol matches handler names case-insensitively, so the web client registers handlers using lower-case names (hub.on('state', …)). When you implement a client, register your handler under the method name; case does not matter. This page lists the canonical server-side constant for each method.

ApplicationHub — /hub/application

The primary hub. On connect it pushes a full snapshot of application state and (redacted) options; thereafter it broadcasts the entire state object again every time any field of the state changes, and the entire options object again every time configuration is reloaded. There are no client→server invocations — it is a one-way push hub.

Server→client methods

MethodWhenPayload
STATEOn connect (to caller), then on every state mutation (broadcast to all)The full State object
OPTIONSOn connect (to caller), then whenever options are reloaded (broadcast to all)The full Options object, redacted — secrets such as passwords and API keys are masked via Options.Redact()

The state snapshot is sent unconditionally on connect; the broadcast is wired in Application.cs via State.OnChange(...)→BroadcastStateAsync and OptionsMonitor.OnChange(...)→BroadcastOptionsAsync. Because the entire object is re-sent on every change, clients should treat each STATE message as a complete replacement, not a delta.

LogsHub — /hub/logs

Streams Serilog log records as they are emitted. On connect, the hub pushes the in-memory ring buffer of recent records so a freshly connected client has immediate context; thereafter each new record is pushed individually.

Server→client methods

MethodWhenPayload
BUFFEROn connect (to caller)Array of LogRecord — the contents of Program.LogBuffer (recent history)
LOGOn every newly emitted log event (broadcast to all)A single LogRecord

The live stream is wired in Application.cs: Program.LogEmitted += (_, log) => LogHub.EmitLogAsync(log), which calls Clients.All.SendAsync("LOG", record).

SearchHub — /hub/search

Streams the lifecycle of searches: the current list on connect, then create/update/delete events for the search records, plus a stream of individual responses as they arrive from the network. This lets a UI render incremental results without polling REST.

Server→client methods

MethodWhenPayload
LISTOn connect (to caller)Array of Search records — the full current list (with responses).
CREATEWhen a new search is started (broadcast)A single Search record with responses set to null.
UPDATEWhen a search's state/counters change, including completion (broadcast)A single Search record with responses set to null.
RESPONSEWhen a new search response is received from a peer (broadcast)An envelope { "searchId": <guid>, "response": <SearchResponse> }.
DELETEWhen a search is deleted (broadcast)A single Search record with responses set to null.
Note

CREATE, UPDATE and DELETE deliberately strip the responses collection (sent as null) to keep broadcasts small — the record carries counters (responseCount, fileCount, state, etc.) but not the result rows. Individual results arrive separately via RESPONSE as they come in. To assemble a full result set, accumulate RESPONSE envelopes keyed by searchId, or fetch GET /api/v0/searches/{id}?includeResponses=true from the REST API. The LIST snapshot on connect is the exception — it includes responses.

There is no completed method; completion is observed as an UPDATE whose state field reaches a terminal value (e.g. Completed, TimedOut). See Search for the Search.state flags enum.

Example: streaming search

const searchHub = createSearchHubConnection(); // {root}/hub/search
searchHub.on('list',   (searches) => seed(searches));            // array, with responses
searchHub.on('create', (search)   => upsert(search));            // responses === null
searchHub.on('update', (search)   => upsert(search));            // responses === null
searchHub.on('delete', (search)   => remove(search.id));
searchHub.on('response', ({ searchId, response }) => append(searchId, response));
await searchHub.start();
// RESPONSE envelope
{
  "searchId": "1f3c9b7e-2a4d-4c1a-9e6b-0a1b2c3d4e5f",
  "response": {
    "username": "someuser",
    "token": 123456,
    "fileCount": 3,
    "files": [ /* File[] */ ],
    "hasFreeUploadSlot": true,
    "queueLength": 0,
    "uploadSpeed": 524288
  }
}

RelayHub — /hub/relay

The relay hub is bidirectional and is the backbone of slskd's distributed (controller/agent) deployment mode. Unlike the other hubs, the relay hub uses a strongly-typed client interface (Hub<IRelayHub>) and a custom challenge/response authentication on top of the SignalR [Authorize]. It is only active when relay.enabled is true and the instance runs in Controller (or Debug) mode; otherwise new connections are aborted on connect. Full coverage — including the file-transfer relay workflow — is in Relay Protocol; this section lists the method signatures for completeness.

Server (controller)→client (agent) methods — IRelayHub

MethodParametersPurpose
Challengetoken: stringSent on connect; the agent must answer via Login with a response derived from the token and the shared secret.
RequestFileInfofilename: string, id: GuidAsk the agent whether a file exists and its length; agent replies with ReturnFileInfo.
RequestFileUploadfilename: string, startOffset: long, id: GuidAsk the agent to stream a file (for serving an upload). Agent uploads via the relay HTTP endpoint, or reports failure via NotifyFileUploadFailed.
NotifyFileDownloadCompletedfilename: string, id: GuidTell the agent a download it requested is complete and ready to be pulled.

Client (agent)→server (controller) invocations

MethodParameters → returnsPurpose
Loginagent: string, challengeResponse: stringAuthenticate the agent: validates the agent name against configured agents, the source IP against the agent's CIDR allow-list, and the challenge response. On failure throws UnauthorizedAccessException.
BeginShareUpload() → GuidObtain a share-upload token to begin uploading the agent's share catalog to the controller. Requires prior Login.
ReturnFileInfoid: Guid, exists: bool, length: longRespond to a RequestFileInfo call.

Authorization

On connect the controller aborts the connection if relay is disabled or the instance is not in Controller/Debug mode. After the SignalR [Authorize] check, the agent must still pass the Challenge→Login handshake; the source IP must fall within the configured per-agent CIDR range. See Relay Protocol and Authentication & Security.

The Events system

The events system is a separate, durable subsystem. An internal EventBus publishes strongly-typed Event records; subscribers include the EventService (which persists each event as an EventRecord row), the WebhookService and ScriptService, and any other internal consumer. Events are surfaced to API clients only through the durable store (poll GET /api/v0/events) and through the push integrations — they are not broadcast over SignalR.

The EventRecord (stored form)

Each raised event is converted to an EventRecord and written to a database table indexed on Timestamp and Type.

FieldTypeDescription
idGuidPrimary key; equals the originating event's Id.
timestampDateTimeUTC time the event occurred; equals the event's Timestamp.
typeEventTypeThe event type, serialized as a string (e.g. "DownloadFileComplete").
datastringThe event-specific payload, serialized as a JSON string (camelCase, nulls retained). The redundant timestamp, type, id and data keys are stripped from this inner object before storage.
Important

In the stored/listed EventRecord, data is a JSON-encoded string, not a nested object. A consumer of GET /api/v0/events must parse record.data a second time to obtain the typed payload. Compare with webhook/script delivery (below), where the event is serialized as a flat JSON object with timestamp/type/id at the top level alongside the payload fields.

Source: src/slskd/Events/Types/EventRecord.cs; src/slskd/Events/EventService.cs

Two endpoints (EventsController), both under policy Any. The events list is the durable, replayable view of the catalog above.

POST/api/v0/events/{type}Any

Raise a sample event of the given type (for testing integrations end-to-end). Webhooks & Scripts

Two built-in event subscribers turn events into outbound side effects. Both subscribe to the bus as catch-all Event subscribers and then filter per configured entry by event type. Each subscriber fires asynchronously and independently — a slow or failing integration does not block the application or other integrations.

Webhooks

Configured under integrations.webhooks as a map of named webhooks. When an event whose type matches a webhook's on list (or the wildcard Any) is raised, the service POSTs the event to the configured URL.

Config keyTypeDefaultDescription
onstring[][]Event type names that trigger this webhook. Use Any to match all.
call.urlstring—Fully qualified URL (must start with http:// or https://).
call.headersarray of {name, value}[]Custom request headers (e.g. an auth token for the receiver).
call.ignoreCertificateErrorsboolfalseSkip TLS certificate validation for the call.
timeoutint325000Per-attempt HTTP timeout in milliseconds (min 500).
retry.attemptsint321Max attempts before giving up; retries use exponential backoff (base 1000 ms, cap 30000 ms). Any non-success status is retried.

The POST body is the event serialized as a single flat JSON object: application/json, camelCase, nulls retained, using the HTML-safe (JavaScriptEncoder.Default) encoder. The event is serialized via its concrete runtime type, so all payload fields are present alongside the base id/timestamp/type/version fields.

POST https://example.com/hook
Content-Type: application/json

{
  "id": "…",
  "timestamp": "2026-06-16T12:34:56.789Z",
  "type": "DownloadFileComplete",
  "version": 0,
  "localFilename": "/downloads/a.mp3",
  "remoteFilename": "@@abcd\\music\\a.mp3",
  "transfer": { /* full Transfer object */ }
}
No payload signature

As of this source, slskd does not sign webhook payloads (no HMAC header). Authenticate the receiver out-of-band by configuring a static secret header under call.headers and validating it server-side, and prefer https.

Scripts

Configured under integrations.scripts as a map of named scripts. When a matching event is raised, the service spawns a process; the full event JSON is passed to the process in the SLSKD_SCRIPT_DATA environment variable (same serialization as webhooks: camelCase, nulls retained, safe encoder).

Config keyTypeDescription
onstring[]Event type names that trigger this script (or Any).
run.commandstring"Command mode": run via the system shell (cmd.exe /c on Windows, $SHELL -c elsewhere). Takes precedence over the other modes.
run.executablestringExecutable to start; defaults to the system shell if omitted.
run.argsstring"Args mode" (default): argument string passed to the executable. Mutually exclusive with arglist.
run.argliststring[]"Args-list mode": list of arguments. Takes precedence over args.

The process runs with WorkingDirectory set to the application's script directory. A non-empty STDERR is treated as failure (logged at Warning); STDOUT is captured and logged at Debug. Scripts read their input like so:

#!/usr/bin/env bash
# $SLSKD_SCRIPT_DATA contains the event JSON
type=$(echo "$SLSKD_SCRIPT_DATA" | jq -r '.type')
file=$(echo "$SLSKD_SCRIPT_DATA" | jq -r '.localFilename')
echo "completed $type: $file"

Connection recipe (language-agnostic)

To consume the application hub from a non-.NET client you must implement (or use a library that implements) the SignalR hub protocol. The sequence below is generic; substitute your platform's SignalR client where one exists (clients exist for JavaScript/TypeScript, Java, Python via signalrcore, Go via community libraries, etc.).

// 1. Obtain a token (see authentication.html):
//      POST /api/v0/session  { "username", "password" }  -> { "token": "<jwt>" }
//    or use a configured API key directly.

// 2. Negotiate. SignalR posts to the hub's /negotiate sub-path.
//    Pass the token as a query parameter so it survives all transports.
POST https://host:5030/hub/application/negotiate?negotiateVersion=1&access_token=<jwt>
//    -> { "connectionId": "...", "connectionToken": "...",
//         "availableTransports": [ { "transport": "WebSockets", ... }, ... ] }

// 3. Open the chosen transport (prefer WebSockets):
ws  wss://host:5030/hub/application?id=<connectionToken>&access_token=<jwt>

// 4. Send the SignalR handshake frame (JSON protocol), terminated by 0x1E:
send  {"protocol":"json","version":1}

// 5. Register handlers. For the JSON hub protocol, an incoming invocation is:
//    {"type":1,"target":"STATE","arguments":[ {...state...} ]}
on "STATE"   -> replace local application state with arguments[0]
on "OPTIONS" -> replace local options with arguments[0]   // redacted

// 6. Respond to server pings (type 6) with a ping, and reconnect with backoff
//    on transport close. On reconnect, the hub re-sends the full STATE+OPTIONS
//    snapshot, so no manual resync is required.
Tip

For durable, exactly-the-data-you-need consumption (build pipelines, notifications), skip SignalR entirely: configure a webhook or script on the event types you care about, or poll GET /api/v0/events and track the newest timestamp you have processed. These survive client downtime; SignalR does not.
